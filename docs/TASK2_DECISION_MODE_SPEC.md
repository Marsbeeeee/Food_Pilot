# FoodPilot Task 2 交互规格

## 文档信息

- 文档主题：Task 2 点单场景输入 MVP 交互规格
- 更新时间：2026-04-05
- 关联文档：`docs/PRODUCT_SPEC.md`
- 适用范围：`frontend/src/pages/Workspace.tsx` 当前主输入区与后续相关 API

---

## 1. 目标

Task 2 的目标不是新做一个独立页面，而是在当前 `Workspace` 主输入区上，以最小 UI 成本增加“点单决策”模式，让用户可以像使用 GPT 的模式切换一样，在同一个输入框内切换不同用途。

该方案的核心收益：

1. 减少额外 UI 面积和前端维护成本。
2. 保留当前用户对 `Workspace` 的主使用习惯。
3. 为后续截图解析、商品对比等能力预留统一入口。
4. 避免“聊天入口”和“点单入口”分裂成两套产品。

---

## 2. 当前前端基线

基于当前 `Workspace` 实现，已有条件如下：

1. 页面底部已经存在一个统一主输入区：单个 `textarea` + 发送按钮。
2. 当前提交链路统一走 `handleSendMessage()`。
3. 现有消息流、会话管理、保存到 Food Log 的逻辑都已围绕该主输入区建立。

因此 Task 2 的实现原则应是：

- 不先拆成新页面
- 不先新做大型表单
- 不先增加复杂多栏布局
- 优先在现有输入区上做轻量增强

---

## 3. 交互原则

### 3.1 单输入区优先

所有用户输入仍优先通过同一个主输入区进入系统。

### 3.2 模式切换优先

不同能力通过“模式切换”区分，而不是通过多入口页面区分。

### 3.3 渐进增强

第一期只支持最必要模式，不一次性把所有能力堆进菜单。

### 3.4 默认保持低学习成本

默认模式仍应是用户最熟悉的普通对话模式，避免老用户进入页面后立即迷失。

---

## 4. 第一阶段建议模式

第一阶段建议只保留 2 个主模式，1 个预留位：

1. `普通对话`
   用于现有聊天、营养问答、自由输入。

2. `点单决策`
   用于商品标题、商品描述、套餐文本的结构化决策输入。

3. `更多`
   暂不展开成完整能力，只作为后续截图解析、商品对比等模式的扩展入口。

第一阶段不建议直接把所有未来能力都做成一级模式，否则会过早增加认知负担。

---

## 5. 推荐交互方案

### 5.1 入口位置

模式切换入口建议放在输入框左侧，与 GPT 截图中的“加号 / 模式入口”类似，作为主输入框的一部分，而不是放到页面 Header 或侧边栏。

建议结构：

- 左侧：模式按钮
- 中间：统一 `textarea`
- 右侧：发送按钮

如果后续加入附件上传，则附件入口应与模式按钮保持同一交互组。

### 5.2 展开方式

点击左侧模式按钮后，弹出轻量菜单或 popover。

第一期菜单建议：

- `普通对话`
- `点单决策`
- `更多`

`更多` 不要求第一期就可用，但 UI 上可预留结构。

### 5.3 选中状态

当前模式必须在输入区内有明确可见的状态，不能只在弹层里显示一次。

推荐做法：

- 模式按钮显示当前模式名
- 或输入框顶部显示一条轻量模式标签

例如：

- `当前模式：普通对话`
- `当前模式：点单决策`

### 5.4 切换后的输入提示

模式切换后，输入框 placeholder 和辅助文案应变化。

示例：

- 普通对话：
  `问营养、问推荐，或直接描述你吃了什么`

- 点单决策：
  `粘贴商品标题、套餐描述，或输入品牌+商品名`

---

## 6. 点单决策模式行为

### 6.1 输入内容

点单决策模式下，第一期支持以下输入：

1. 商品标题
   例如：`霸王茶姬 伯牙绝弦 大杯 三分糖`

2. 商品描述
   例如：`麦当劳 双人套餐，两个汉堡+两份薯条+两杯可乐`

3. 品牌 + 商品名
   例如：`瑞幸 生椰拿铁`

### 6.2 与普通对话的差异

普通对话模式：

- 允许自然语言问句
- 默认走当前聊天链路

点单决策模式：

- 优先把输入当成“商品对象”
- 后端优先走商品理解、品牌映射、决策卡片链路
- 即使用户没有写完整问句，也应尝试解析

### 6.3 提交行为

点单决策模式提交后，不要求用户先创建独立“分析表单”。

仍然沿用当前发送动作：

- Enter 发送
- Shift + Enter 换行
- 点击发送按钮提交

---

## 7. 模式切换对消息流的影响

### 7.1 会话层

第一阶段不建议把不同模式拆成不同会话类型。

仍使用现有会话结构，但每条用户消息应允许带上 `mode` 元数据。

### 7.2 消息展示层

用户消息可以继续按普通气泡显示，但建议在后续版本中为点单决策模式增加轻量来源标记，例如：

- `Decision`
- `点单决策`

第一阶段不是强制项。

### 7.3 Assistant 返回

当模式为 `点单决策` 时，assistant 不应优先返回普通文本解释，而应优先返回统一 `decision_card` 结构。

---

## 8. 请求契约建议

第一阶段无需废弃现有聊天接口，但请求体应具备承载模式信息的能力。

建议在用户消息请求中增加：

```json
{
  "message": "霸王茶姬 伯牙绝弦 大杯 三分糖",
  "mode": "decision",
  "profileId": 123
}
```

如果未来加入更强场景输入，也可扩展为：

```json
{
  "mode": "decision",
  "title": "霸王茶姬 伯牙绝弦 大杯 三分糖",
  "description": "",
  "sourcePlatform": "taobao_flash_buy",
  "profileId": 123
}
```

第一阶段建议：

1. 前端继续使用统一发送动作。
2. 后端依据 `mode` 决定分流到普通 chat 还是 decision 流程。
3. 不要同时维护两套完全分离的提交接口，除非后端复杂度已经证明必须拆开。

---

## 9. 状态设计

### 9.1 默认状态

- 默认进入 `普通对话`
- 保持和现有体验一致

### 9.2 模式记忆

第一阶段建议按本地会话记忆最近一次选择的模式，但不跨设备强同步。

原因：

- 用户通常会连续使用同一种模式
- 本地记忆能减少反复切换
- 跨端同步不是当前阻塞项

### 9.3 错误状态

点单决策模式下，常见错误应有更明确提示：

1. 输入为空
2. 促销文案过多，缺少商品主体
3. 品牌命中但商品未命中
4. 商品和套餐信息不完整

错误提示应优先告诉用户“缺什么”，而不是统一提示“请补充更多信息”。

---

## 10. 与截图入口的关系

第一阶段虽然不要求完整截图模式上线，但模式切换方案必须为它留接口。

建议：

1. 左侧模式入口和附件入口放在同一区域。
2. 后续加入截图解析时，不再新开一块输入区，而是：
   - 先切到 `点单决策`
   - 再通过附件入口上传截图
3. 截图识别完成后，仍回填到同一个输入链路中。

这样可以保证文本输入和截图输入最终汇聚到同一任务流。

---

## 11. 第一阶段非目标

以下事项第一阶段不做：

1. 不单独新建完整“点单决策页面”
2. 不引入复杂多步骤表单
3. 不在第一期就暴露过多模式
4. 不做完整附件中心
5. 不做多模式多线程会话体系

---

## 12. 开发拆解建议

### 前端

1. 在 `Workspace` 主输入区增加模式按钮和轻量菜单。
2. 增加模式状态管理，例如 `chat | decision`。
3. 根据模式切换 placeholder、辅助文案和发送行为元数据。
4. 预留后续附件入口位置。

### 后端

1. 在消息请求中支持 `mode` 字段。
2. 根据 `mode` 分流到现有 chat 链路或新的 decision 链路。
3. 对 `decision` 模式返回统一结构化结果。

### 测试

1. 前端测试模式切换状态是否正确影响提交参数。
2. 前端测试 placeholder / 文案是否随模式变化。
3. 后端测试 `mode=decision` 时是否进入正确分流。
4. 回归现有普通对话模式，保证老链路不被破坏。

---

## 13. 验收标准

满足以下条件即可认为 Task 2 的交互 MVP 成立：

1. 当前 `Workspace` 输入框支持模式切换。
2. 用户能在同一输入区中切到 `点单决策` 模式。
3. 切换后 placeholder、提示文案和请求参数会变化。
4. 提交后后端可识别 `decision` 模式并走对应链路。
5. 不需要新建独立页面或大块新 UI 也能完成 MVP。
