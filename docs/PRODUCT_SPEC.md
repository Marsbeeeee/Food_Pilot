## 产品方向总述

FoodPilot 的长期目标，不再仅仅是一个独立的食物热量分析网站，而是一个面向**外卖、即时零售与电商点单场景**的 AI 饮食决策助手。产品希望在用户实际做出购买决定之前，帮助用户更快理解商品的大致营养结构、潜在饮食风险，以及该商品是否适合当前用户的个人目标。

现阶段，FoodPilot 的核心定位应从“分析工具”升级为“决策辅助系统”。这意味着产品不只回答“这个食物有多少热量”，而是进一步回答“这个商品对我来说是否值得点”“有没有更合适的替代选择”“如果我想继续点，应该如何调整搭配”。

考虑到淘宝闪购 App 等平台的接入权限存在较高不确定性，FoodPilot 当前阶段应优先采用**外部场景助手**的产品形态，即不强依赖平台内部插件入口，而是围绕这些平台的真实使用场景，为用户提供可独立使用的决策支持能力。用户可以通过输入商品标题、粘贴商品描述、上传商品截图等方式，把淘宝闪购、外卖平台或即时零售场景中的商品带入 FoodPilot，再由系统完成解析、估算与建议输出。

## 一、产品定位

FoodPilot 是一个面向外卖与即时零售场景的 AI 饮食决策助手，帮助用户在点单前快速理解商品结构，估算营养组成，并结合个体目标给出是否推荐、风险提示和替代建议。

与传统热量查询工具不同，FoodPilot 的核心价值不在于提供单次静态估算，而在于提供**品牌感知、商品理解、个体适配、点单决策**四层合一的智能辅助能力。

---

## 二、核心产品目标

### 2.1 总体目标

将 FoodPilot 从“手动输入食物并获得热量分析”的网站，逐步升级为“可服务真实点单场景的决策系统”，使其能够在用户看到某个外卖或商品时，快速帮助用户判断是否适合自己。

### 2.2 近期目标

在不依赖淘宝闪购等平台官方插件接入能力的前提下，先完成一套可独立运行的“场景输入到决策输出”流程，验证用户在真实点单场景中的使用价值。

### 2.3 中期目标

围绕高频品牌、高频商品类型和高频用户目标，逐步沉淀标准化商品映射、品牌模板和个体决策规则，使 FoodPilot 的结果从“泛化估算”提升到“品牌级合理估算 + 用户级适配判断”。

### 2.4 长期目标

在条件允许时，将 FoodPilot 进一步发展为可接入更多真实服务入口的饮食决策 Agent，包括但不限于浏览器插件、外部分享入口、小程序或潜在平台合作场景。

## 三、目标用户与使用场景

### 3.1 目标用户

FoodPilot 当前重点服务以下用户群体：

1. 有减脂、控糖、增肌、维持体重等明确饮食目标的用户
2. 经常点外卖、喝饮品、购买即时零售商品的用户
3. 需要在多个商品之间快速做出选择，但缺乏直观判断依据的用户
4. 希望在不进行复杂记录的情况下，先获得点单层面建议的轻量用户

### 3.2 典型使用场景

#### 场景 A：点单前快速判断

用户在淘宝闪购、外卖平台或其他即时零售页面看到某商品时，希望知道该商品的大致热量、主要风险以及是否适合当前自己的目标。

#### 场景 B：多个商品之间对比决策

用户在多个候选商品之间犹豫，希望知道哪一个更适合自己，或哪一个更容易通过简单改配变得更合理。

#### 场景 C：品牌相似商品差异判断

用户面对不同品牌的同类商品，例如不同品牌的奶茶、汉堡或便当，希望知道它们之间是否存在明显营养差异。

#### 场景 D：个体化适配判断

用户不只希望得到统一的“高热量”结论，而是希望系统结合自己的目标、热量预算和偏好，给出更个体化的建议。

## 四、产品形态与入口设计

### 4.1 当前阶段产品形态

当前阶段，FoodPilot 以“外部场景助手”为主要形态。系统不强依赖目标平台内部注入或官方插件入口，而是通过外部独立页面承接用户需求。

### 4.2 主要输入入口

FoodPilot 应优先支持以下输入方式：

1. 商品标题输入
   用户直接复制商品名称或套餐名称进行查询。

2. 商品描述粘贴
   用户将商品详情、套餐说明或页面文本粘贴到系统中。

3. 商品截图上传
   用户上传来自淘宝闪购、外卖平台或即时零售页面的截图，由系统识别并解析商品信息。

4. 后续扩展入口
   未来可继续扩展为商品链接解析、分享至 FoodPilot、PC 网页插件等方式。

### 4.3 输出形式

系统输出不应停留在单纯的热量数字，而应形成更接近决策卡片的结果结构，包括：

1. 商品估算结果
2. 风险提示
3. 推荐等级
4. 个体适配说明
5. 可执行替代建议

## 五、核心能力框架

FoodPilot 的能力框架可分为五层。

### 5.1 场景输入层

该层负责承接来自真实点单场景的原始输入，并将其转化为可供系统进一步处理的结构化信息。

输入来源包括商品标题、商品描述和商品截图。该层的目标不是直接做营养判断，而是尽可能还原商品的关键信息，包括品牌、商品类型、规格、附加项、套餐组成等。

### 5.2 商品理解层

该层负责把用户输入的商品文本或截图内容解析成可计算的商品结构。

该层在面向真实外卖、电商与即时零售平台演进时，不应只停留在“临时解析结果”，而应逐步沉淀到可复用的**分层商品目录结构**。当前建议的基础层级为：

1. 第一层：菜系 / 品类层
   例如奶茶、咖啡、汉堡、炸鸡、盖饭、便当、轻食、零食、烘焙、早餐组合等。

2. 第二层：品牌层
   例如麦当劳、肯德基、喜茶、霸王茶姬、瑞幸、库迪、塔斯汀等。

品牌层在当前阶段不追求全量覆盖，优先顺序应明确倾向于**有一定体量、SKU 相对稳定、用户认知强、真实点单频次高的连锁品牌**。原因是这类品牌更适合优先建立标准商品映射、品牌模板和可复用估算规则。

3. 第三层：品牌商品层
   即品牌下的具体产品，例如某品牌的生椰拿铁、香辣鸡腿堡、双拼便当、珍珠奶茶等。

这个分层结构的作用不只是为了展示，而是为了同时支撑：

- 商品理解与标准化映射
- 品牌模板估算
- 用户历史记录归档
- 收藏与复用
- 后续搜索与召回能力

需要强调的是：FoodPilot 不应把“有品牌商品”和“可估算商品”画等号。对于无品牌菜品、自制菜品或用户只知道菜名但不知道品牌的场景，系统仍应允许进入估算链路，只是其对象会更多依赖品类层、做法特征和澄清补问，而不是依赖品牌商品层。

核心任务包括：

1. 品牌识别
   识别商品属于哪个品牌或商家体系，例如麦当劳、肯德基、喜茶、霸王茶姬、塔斯汀等。

2. 商品标准化
   将原始标题映射为标准化商品类型，例如“香辣鸡腿堡”“生椰拿铁”“珍珠奶茶”“双拼便当”等。

3. 规格识别
   提取中杯、大杯、小份、大份、加量、双人套餐等信息。

4. 配置项识别
   提取少糖、无糖、去奶盖、加珍珠、加芝士、加蛋、换饮品等附加项。

5. 套餐拆分
   将复杂套餐拆分为主商品、配餐、饮品和附加项，为后续估算提供更准确输入。

### 5.3 品牌感知估算层

该层负责基于品牌、商品类别和配置项，对商品的营养值进行合理估算。

系统不应仅按通用菜名估算，而应尽量做到品牌感知。例如，不同品牌的同类鸡腿堡、奶茶或便当，其基础模板应当不同。系统可通过“品牌模板 + 配置修正”的方式得到最终估算值。

该层输出内容包括：

1. 估算热量
2. 估算宏量营养素，如蛋白质、脂肪、碳水
3. 主要热量来源判断
4. 当前估算的置信度等级

### 5.4 个体化决策层

该层负责将商品估算结果与用户个人背景结合，判断该商品是否适合当前用户。

该层的判断原则不应是“一旦有风险就要求用户立刻改掉”，而应强调**长期可持续性**。FoodPilot 需要承认饮食除了目标管理，也包含情绪满足、社交和快乐价值。因此系统在做判断时，应优先帮助用户在“不打破整体节奏”的前提下做更好的选择，而不是一味输出克制和劝退。

决策时应参考以下信息：

1. 用户目标，例如减脂、维持、增肌
2. 用户热量目标
3. 用户当前已摄入或剩余热量预算
4. 饮食偏好与忌口
5. 潜在过敏信息
6. 用户对高糖、高油、高碳等风险的容忍度

该层输出应包括：

1. 推荐等级，例如推荐、谨慎、暂不推荐
2. 推荐原因
3. 风险标签
4. 简要解释
5. 节奏友好说明

“节奏友好说明”用于告诉用户：当前商品是否可以作为一次可接受的享受、是否需要在当天其他餐次稍作平衡、以及是否无需因为一次选择产生过强负罪感。

### 5.5 替代与调整建议层

该层负责在用户不想完全放弃商品时，给出更可执行的替代方案。

建议形式包括：

1. 同商品内部改配建议
   例如将含糖饮品改为无糖，将大份改为中份，去除高热量加料等。

2. 套餐结构优化建议
   例如保留主食，替换薯条或甜饮。

3. 品类级替代建议
   例如推荐更适合当前目标的便当、轻食、饮品或品牌同类选项。

4. 快乐保留建议
   例如保留最想吃的主体商品，只调整配餐、糖度、份量或当天其他搭配。

该层的目标不是做全面推荐系统，而是帮助用户在“仍然想点”的前提下，找到更合理的选择路径。系统的建议应默认优先寻找“保留快乐 + 控制代价”的折中方案，而不是默认要求用户放弃当前选择。

## 六、关键差异化方向

### 6.1 品牌感知型估算

同类商品在不同品牌之间存在显著差异，FoodPilot 需要避免仅按通用菜名给出统一数值，而应逐步建立品牌级模板，使结果更贴近真实消费场景。

### 6.2 个体化适配结论

FoodPilot 的输出不应是对所有用户统一的判断。同一商品面对不同目标、不同剩余热量和不同偏好的用户，应给出不同结论。这是产品从“估算工具”升级为“决策系统”的核心标志。

### 6.3 场景化输入支持

用户在真实点单场景中不会主动填写标准化营养信息，因此系统必须围绕截图、标题和套餐描述等低结构化输入进行设计，尽可能降低使用门槛。

### 6.4 可执行而非抽象的建议

相比只提示“热量较高”，FoodPilot 更应输出用户可以立即采取的行动建议，例如换饮品、减配料、改份量、换品类等。

## 七、系统数据与能力沉淀方向

### 7.1 分层商品目录层

系统需要逐步建立一套标准化商品知识体系，用于支撑不同品牌、不同标题和不同配置项的统一理解与映射。

目录结构不应是扁平商品列表，而应优先采用三层组织：

1. 第一层：菜系 / 品类
2. 第二层：品牌
3. 第三层：品牌商品

采用该结构的原因包括：

- 更符合真实点单平台的浏览与决策方式
- 便于把“同类比较”和“同品牌比较”分开处理
- 便于品牌模板、配置修正规则和商品展示资产挂载
- 便于用户记录、收藏和历史回看时保持稳定归档

这一层应包含但不限于以下字段：

- `category_id / category_name`
- `brand_id / brand_name`
- `product_id / product_name`
- `aliases`
- `default_specs`
- `default_components`
- `supported_options`
- `status`
- `source_scope`（平台公共 / 品牌公共 / 用户私有）

### 7.2 用户私有补充层

在真实平台场景中，不可能一开始就录全所有品牌商品。因此系统必须允许用户在“未命中标准商品”时，补充自己的私有商品记录。

该能力建议采用“用户私有收藏夹 / 私有商品库”的形式，而不是直接写入全局公共目录。

其基本规则应为：

1. 当系统未命中品牌商品层，或仅命中到品牌但无法确定具体商品时，允许用户自行补录。
2. 用户补录的商品默认只进入该用户自己的私有收藏夹，不直接污染公共库。
3. 用户补录时，仍应尽量挂靠到已存在的第一层品类和第二层品牌，减少数据碎片。
4. 若连品牌也未命中，则允许用户创建“未收录品牌下的私有商品”。
5. 用户私有商品可以被该用户后续搜索、复用、比较和保存，但不默认参与公共模板训练。

用户私有补充层的目标不是替代公共商品库，而是解决真实平台商品更新快、SKU 多、长尾商品多的覆盖问题。

### 7.3 品牌模板层

针对高频品牌和高频商品，逐步建立品牌级基础估算模板，提升结果一致性和可信度。

品牌模板的首批建设对象，应优先是全国性或区域性有一定体量的连锁品牌，而不是零散单店或临时活动 SKU。优先级判断建议参考：

1. 连锁规模和市场覆盖
2. 商品标准化程度
3. 菜单稳定性
4. 用户点单频率
5. 是否能形成高复用模板

### 7.4 配置修正规则层

为常见规格变化和附加项建立修正规则，例如糖度变化、小料增加、套餐替换等，以支持更细粒度估算。

### 7.5 决策规则层

沉淀与用户目标相关的判断逻辑，使系统不仅能估算，还能稳定输出适配结论和建议。

### 7.6 结果置信度机制

系统应明确区分高置信度、中置信度和低置信度结果。当输入过于模糊或套餐信息不足时，应降低结果精度并提示用户当前结论仅供参考。

### 7.7 搜索与召回层

在分层商品目录建立后，系统还需要一套与之匹配的搜索与召回能力，用于支持：

- 标题快速命中
- 品牌内商品召回
- 用户私有收藏夹召回
- 同品类替代商品召回

但搜索不是当前阶段最优先的阻塞项。当前阶段应先把目录层级、商品映射和用户私有补充能力做好，再在其上迭代搜索精度。否则会出现“搜索做得复杂，但底层目录不稳定”的问题。

## 八、阶段性实施路径

### 8.1 第一阶段：完成基础场景闭环

目标是在当前网站基础上，完成从“商品输入”到“决策输出”的最小可用流程。

这一阶段的重点包括：

1. 支持商品标题输入
2. 初步支持商品截图识别
3. 建立基础分层商品目录，至少明确品类层、品牌层、品牌商品层
4. 完成商品品牌识别与基础标准化
5. 支持未收录商品进入用户私有收藏夹
6. 输出推荐等级、风险提示和替代建议
7. 建立高频品类的基础模板，例如奶茶、汉堡、便当、炸物、咖啡

### 8.2 第二阶段：增强品牌与配置精度

目标是让系统从泛化估算提升为更稳定的品牌感知型估算。

这一阶段的重点包括：

1. 增加高频品牌模板
2. 增加糖度、规格、小料、套餐附加项识别
3. 提升复杂商品标题与套餐解析能力
4. 增加结果置信度标识
5. 建立用户私有商品与公共目录之间的弱映射关系
6. 优化替代建议的可执行性

### 8.3 第三阶段：强化个体化与长期使用价值

目标是让 FoodPilot 不只判断单次商品，而是逐步形成更个体化的长期辅助能力。

这一阶段的重点包括：

1. 更好结合用户 profile 做个体化决策
2. 结合 Food Log 和历史选择形成上下文判断
3. 提升同类商品对比能力
4. 基于分层目录与用户私有收藏夹增强历史回看与复用效率
5. 逐步补强搜索与召回算法
6. 支持更多场景入口，例如链接解析、分享入口或 PC 插件

### 8.4 第四阶段：探索平台或生态接入

在产品能力与用户价值已被验证后，再探索更深的入口形态，包括浏览器插件、小程序、平台合作或其他生态整合方式。

该阶段不应作为当前产品成败的前提，而应视为后续可选扩展方向。

## 九、当前阶段的产品边界

### 9.1 不追求医疗级准确性

FoodPilot 当前目标是提供合理、可解释、对点单决策有帮助的营养估算与建议，而非医疗或实验室级精确测量。

### 9.2 不依赖平台内部插件权限

当前阶段不将淘宝闪购 App 内部插件接入作为必须条件。产品优先验证用户价值与核心能力，再视条件探索更深层接入。

### 9.3 不追求全平台全商品覆盖

系统应优先集中于高频品牌、高频品类和高频用户决策场景，通过逐步扩展提升覆盖范围，而不是一开始追求全量覆盖。

### 9.4 不把输出停留在静态数值

FoodPilot 的核心不是仅返回热量数字，而是给出决策支持，因此 UI、文案和返回结构都应围绕“帮助用户做选择”来设计。

## 11. 当前版本定位与实现范围

- 自动饮食日记

## 14. 功能规格

### 4.1 Auth

后端接口：

- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`
- `DELETE /auth/me`

当前约束：

- 注册需要合法邮箱、非空 `displayName`、至少 8 位密码。
- 使用 Bearer Token。
- Token 为后端自签 HMAC JWT，包含 `sub / iat / exp`。
- 前端会将 token 与当前用户信息保存在本地。
- 删除账号后，数据库通过外键级联删除该用户相关 Profile、Chat、Food Log、Insights 数据。

### 4.2 Chat / Assistant

#### 4.2.1 会话能力

后端接口：

- `POST /chat/sessions`
- `GET /chat/sessions`
- `GET /chat/sessions/{session_id}`
- `PATCH /chat/sessions/{session_id}`
- `DELETE /chat/sessions/{session_id}`
- `POST /chat/messages`
- `POST /chat/sessions/{session_id}/messages`

当前行为：

- 支持新建空会话。
- 支持在空会话中直接发首条消息并自动生成会话。
- 首条用户消息会用于更新默认标题。
- 删除会话为永久删除，不做软删除。
- 删除会话后，Food Log 记录仍保留，但关联的 `session_id` 会被清空，前端展示为 `Source Chat Deleted`。

#### 4.2.2 消息类型契约

对外消息类型统一为：

- `text`
- `meal_estimate`
- `meal_recommendation`

兼容规则：

- 数据库内部仍可能存储历史值 `estimate_result`
- Router / Schema 层统一映射为 `meal_estimate`

#### 4.2.3 意图分流

当前实现是“规则分流”，不是 LLM 分类器。

分流结果：

- `meal_estimate`：命中“多少热量 / 多少蛋白质 / 营养怎么样 / 大概多少”等估算关键词
- `meal_recommendation`：命中“推荐 / 替代 / 选哪个 / 训练后吃什么 / 怎么优化”等推荐关键词
- `text`：命中解释、说明、区别、为什么等文本类关键词
- `_clarification`：输入模糊、像问题但又不像具体食物描述时，先返回澄清提问

补充说明：

- 估算优先级高于推荐。
- 解释型追问会优先走 `text`，避免把“为什么推荐这个”误判成新的推荐请求。
- 当用户输入触发了估算意图，但菜品标准化匹配置信度不足时，Chat 应优先进入 `_clarification` 追问，而不是直接生成结构化估算结果。
- 典型场景包括“炒面热量”“盖饭热量”“套餐多少卡”这类标准菜品指向不清、缺少关键修饰词或可能对应多个标准菜的表达。
- 对于“用户不知道品牌”“家里自己做的菜”“只有菜名没有做法”的情况，Chat 也不应直接拒绝，而应通过有针对性的追问补齐影响估算差异的关键信息，例如：
  - 做法差异
  - 口味倾向
  - 是否带糖/酱/浇头
  - 主要食材比例
  - 大致份量或餐具大小
- 这类追问应尽量具体，避免泛泛地只说“请补充更多信息”。例如可以问“番茄炒蛋偏甜口还是偏咸口”“是自己家常炒的还是餐馆重油做法”“一盘大概几个人分量”等。
- 当前规则对高频中文问法已有回归测试覆盖，但复杂表述仍存在误分流风险。

#### 4.2.4 Assistant 输出边界

- Chat 内估算结果不会自动写入 Food Log。
- Chat 内推荐结果不能直接保存到 Food Log。
- 多食物估算时，消息 `payload.estimates[]` 会保留每个子估算块，前端可逐块展示。
- Chat 估算结果采用“两层组织逻辑”：
  1. 最外层先判断用户是否描述了多个不同菜品或独立食物。
  2. 若是，则按“一个菜品 / 独立食物一张卡”拆开展示；若否，则只展示一张卡。
- 在每一张菜品卡片内部，都应展示当前菜品的组成项明细、各组成项对应的热量与三大营养素，以及该菜品总热量。
- Chat 估算结果的“拆卡”粒度是“菜品 / 独立食物”，不是单个配料或烹调用料。
- 当用户在同一句里明确提到多个菜品或独立食物时，例如“番茄炒蛋和肉末茄子”或“辣椒炒肉和白米饭”，应拆成多个估算卡片。
- 当用户问的是一个单独菜品时，例如“肉末茄子热量多少”，即使系统需要展示茄子、肉末、植物油等组成项，也必须保留为一个菜品卡片，并在该卡片内部按食材分行展示。
- 单菜品内部的食材拆解属于卡片内明细，不得提升为 `payload.estimates[]` 的并列卡片。
- 对单菜品估算，优先输出“菜名 + 组成项表格”的结构，效果应接近“番茄炒蛋盖饭 -> 白米饭 / 鸡蛋 / 番茄 / 烹饪用油”这种单卡内逐行拆解。
- 对多菜品估算，`payload.estimates[]` 中的每个 estimate block 都应对应一个单独菜品卡片；每个 block 自身的 `items[]` 应优先是该菜品的组成项表格，而不是只有一条聚合总项。
- 若标准菜品匹配仍处于低置信状态，Chat 不应直接输出可保存的信息卡片，而应先追问用户补足关键信息，如做法、主配料、甜度、品牌、是否带浇头等。
- 若用户明确表示“不知道品牌”或“是自己做的”，系统应继续尝试以品类模板 / 家常做法模板估算，而不是因为缺少品牌字段就中断能力；只是此时必须通过澄清问题提升估算质量，并在结果中降低置信度。

#### 4.2.5 推荐安全约束

当前实现：

- 当请求携带 `profileId` 时，推荐链路会读取 Profile。
- 如果 Profile 中存在 `allergies`，系统会对推荐结果做关键词级过敏原拦截。
- 命中过敏原时，不返回原推荐，而返回“推荐已拦截”的安全提示。

当前边界：

- 这是关键词级拦截，不是结构化配料理解。

#### 4.2.6 Chat 估算卡片层级强约束（防回归）

适用范围：

- Chat 中所有 `meal_estimate` 消息的结构组织与前端渲染。
- 包括 `payload.items[]` 单卡模式与 `payload.estimates[]` 多卡模式。

目标：

- 统一为“菜品 / 独立食物是最外层卡片；组成食材与营养素在卡片内部展示”。
- 防止再次出现“把一个菜拆成多个外层卡片”的错误。
- 在不改变当前估算卡片 UI 形态（卡片样式、表格结构、保存区布局）的前提下，保证“多菜品输入 = 多张并列外层卡片”。

术语定义：

- `外层卡片（dish card）`：用户一次描述中的一个“直接食用对象”（菜品或独立食物/饮品）。
- `内部组成项（ingredient row）`：该外层卡片内部的食材/配料行，按行展示份量与营养素。
- `顶层拆卡`：`payload.estimates[]` 的并列 block 数量。

强约束（MUST）：

1. 顶层拆卡单位必须是“菜品 / 独立食物”，不能是食材。
2. 单菜品输入必须只生成 `1` 张外层卡片。
3. 单菜品内部的拆解（如猪肉、青椒、植物油）必须仅出现在该卡片 `items[]` 中。
4. 多菜品输入时，`payload.estimates[]` 每个 block 对应一个菜品或独立食物；不得把同一菜品内部食材提升为并列 block。
5. 对“牛肉汉堡”这类复合菜名，默认仍是 `1` 张“牛肉汉堡”外层卡；其面包、肉饼、酱料是内部组成项，不得成为并列外层卡。
6. 只有当用户明确把组成部分作为“分别吃的独立对象”表达时，才允许拆成多个外层卡（例如“我吃了牛肉饼和白面包”）。
7. 若当前无法稳定判断用户说的是一个标准菜还是多个独立对象，必须走澄清提问，不得直接输出错误层级卡片。
8. 当用户一次输入多个菜品/饮品时，前端必须按既有卡片 UI 重复渲染多个同结构卡片（每个菜品一张），禁止把多个菜品合并为一张“组合总卡”。

禁止模式（MUST NOT）：

- 用户问“牛肉汉堡热量”时，输出两个并列外层卡：“瘦牛肉饼” + “汉堡面包（白面包）”。
- 用户问“辣椒炒肉热量”时，把“辣椒”“猪肉”“油”作为三张并列外层卡。
- 用食材名替代菜名作为单菜品外层卡标题（除非用户明确只问该食材本身）。
- 用户问“今天中午吃了牛肉汉堡和一杯柚子苏打水”时，仅输出一张“牛肉汉堡配柚子苏打水”合并卡。

数据契约要求（Chat payload）：

- 单菜品模式：
  - `payload.estimates[]` 可为空或不存在。
  - 外层卡使用 `payload.title + payload.items[]` 渲染。
  - 若存在 `itemization_mode = single_dish_ingredients`，前端必须强制按“单外层卡 + 内部组成项”渲染，不得再拆卡。
- 多菜品模式：
  - `payload.estimates[]` 必须存在且每个 block 是一个菜品/独立食物。
  - 每个 block 的 `items[]` 表示该菜品内部组成项，而非新的顶层拆卡候选。

标准示例（应当通过）：

- 输入：“辣椒炒肉热量多少”
  - 外层卡片数：`1`
  - 外层卡片标题：`辣椒炒肉`
  - 内部组成项：`猪瘦肉/青椒/植物油...`
- 输入：“今天中午吃了牛肉汉堡和一杯苏打水”
  - 外层卡片数：`2`
  - 外层卡片标题：`牛肉汉堡`、`苏打水`
  - `牛肉汉堡` 卡内可有：`汉堡面包/牛肉饼/酱料...`
- 输入：“今天中午吃了牛肉汉堡和一杯柚子苏打水”
  - 外层卡片数：`2`
  - 外层卡片标题：`牛肉汉堡`、`柚子苏打水`
  - 不允许标题聚合为 `牛肉汉堡配柚子苏打水`
- 输入：“我吃了牛肉饼和白面包”
  - 外层卡片数：`2`
  - 外层卡片标题：`牛肉饼`、`白面包`

回归验收清单（每次改动 Chat 估算展示逻辑时必测）：

1. 单菜品必须保持单外层卡。
2. 单菜品内部组成项必须保留在卡内表格，不得升为并列外层卡。
3. “复合菜名（如汉堡/盖饭/炒饭）”不得被错误拆为其内部配料外层卡。
4. 多菜品输入必须一菜一卡，且每卡内部可继续展示组成项。
5. 低置信场景优先澄清，不输出错误层级的可保存结果卡。
6. 在多菜品场景下，保持现有单卡 UI 样式不变，仅增加卡片数量，不新增“组合汇总卡”替代逐菜卡。

下一阶段新增约束：

- 每张可保存菜品卡片都必须带明确的分类标签，至少包括：
  - `category_id / category_name`
  - `brand_id / brand_name`
  - `product_id / product_name`
  - `product_scope`（公共商品 / 用户私有商品）
  - `item_role`（主商品 / 饮品 / 配餐 / 套餐子项）
  - `save_container_key`
- 这些标签不仅服务于展示，还必须作为后续保存、归档和容器路由的依据。
- 若系统当前无法给出稳定分类标签，则应先澄清或降级，而不是生成一个可保存但无法正确归档的卡片。

### 4.3 独立 Estimate API（系统能力层）

后端接口：

- `POST /estimate`

#### 4.3.1 产品定位

`/estimate` 是系统级的结构化营养估算能力接口，不作为当前版本的用户主入口。

其主要职责：

- 为 Chat 提供底层估算能力
- 为 Food Log 提供结构化数据来源
- 为未来快速录入、批量估算和外部接入提供能力基础

#### 4.3.2 与 Chat 的关系

- Chat 是用户主入口，负责意图理解、澄清和结果组织。
- `/estimate` 是原子能力层，仅负责单次估算计算。
- Chat 在命中 `meal_estimate` 时，应通过统一估算链路调用 `/estimate` 对应能力，而不是发展独立估算逻辑分支。
- 当菜品标准化匹配置信度不足时，Chat 先负责澄清追问；只有在补足信息、达到可稳定映射标准菜品的条件后，才继续进入结构化估算与后续保存链路。
- 对于无品牌或自制菜品，Chat 的目标不是强行映射品牌商品，而是尽量通过澄清把对象稳定映射到品类模板、做法模板或通用估算模板，再继续进入结构化估算链路。

#### 4.3.3 当前入口策略

- 当前版本不提供独立 Estimate 页面或主按钮入口。
- 所有用户估算行为优先通过 Chat 触发。
- `/estimate` 仅作为内部能力和前端 client 保留。

#### 4.3.4 后续演进方向（非当前版本）

未来可能扩展为：

- Quick Add（无对话快速录入）
- 批量食物估算入口
- 第三方 API / 多端复用能力
- 面向中国用户的标准化食物数据底座

其中“面向中国用户的标准化食物数据底座”下一阶段的产品要求为：

- 继续扩充中国高频菜品、早餐、外卖套餐、饮品、便利店与健身餐覆盖。
- 为知识条目建立稳定 `food_id`、标准名、中文别名、份量提示、营养值来源与更新时间。
- 优先提升“中文真实菜品匹配与估算稳定性”，而不是先追求通用国际食物大全。
- 数据形态允许继续以离线版本化数据集为主；是否迁移为独立表或独立数据库，取决于后续后台运营、批量维护与多来源同步需求。

#### 4.3.5 不做事项（当前阶段）

- 不作为 Chat 的替代入口
- 不提供与 Chat 重叠的推荐或解释能力
- 不在 UI 中形成与 Chat 并列的主功能模块

#### 4.3.6 请求与返回结构

请求支持：

- `query`
- `clientRequestId`
- `profileId`
- `sessionId`

返回结构包含：

- `success`
- `data`
- `error`
- `clientRequestId`
- `foodLogId`
- `saveStatus`

补充约束：

- 多菜品场景下，`data.items[]` 的语义是“顶层菜品 / 独立食物项”，Chat 会进一步拆成多个卡片。
- Chat 在将多菜品结果拆成多个卡片后，应继续逐卡尝试补齐该菜品的组成项明细，使每张卡都能展示“组成项行 + 该菜品总量”。
- 单菜品场景下，`data.items[]` 可以直接承载该菜品的组成项明细；此时结果必须保留为一个卡片，并通过 `itemization_mode = single_dish_ingredients` 约束前端不要再拆卡。
- 若命中的是单个标准菜品，系统应优先输出可直接渲染为单卡表格的组成项明细，而不是只返回一条总项。

### 4.4 Food Log

#### 4.4.1 产品定位

Food Log 是“显式保存的结果库”，不是自动记录所有饮食行为的流水账。

但在下一阶段，Food Log 不应继续停留在“简单条目列表”层面，而应逐步演化为与分层商品目录对齐的结构化记录层。也就是说，一条记录未来应尽量能回答：

1. 它属于哪个品类 / 菜系
2. 它属于哪个品牌
3. 它对应品牌下的哪个具体商品
4. 它是公共商品还是用户私有商品
5. 它是原商品、改配商品还是套餐拆分结果

这样做的目的不是把 Food Log 做复杂，而是为了让真实平台场景下的记录、收藏、比较、复用和后续分析都建立在稳定对象之上，而不是建立在一次性的自由文本上。

#### 4.4.1A 容器模型（下一阶段补充）

详细数据模型规格见：`docs/CONTAINER_DATA_MODEL_SPEC.md`

下一阶段不应把所有保存结果都继续放进同一种平铺列表，而应引入明确的容器语义。

建议至少存在以下三类容器：

1. 正式分类容器
   与分层商品目录对齐，按“品类 -> 品牌 -> 品牌商品”或“品类 -> 私有商品”归档。

2. 未明确来源容器
   用于承接“可以估算、可以保存，但来源或归档标签暂不够稳定”的菜品。
   典型场景包括：
   - 用户不知道品牌
   - 家里自制菜
   - 路边店 / 小店菜品
   - 只能确认菜名，但不能稳定映射到具体品牌商品

3. 收藏夹容器
   沿用当前收藏夹能力，但语义收紧为“用户希望后续复用、回看、比较或再次选择的对象集合”。
   收藏夹不是正式营养分析容器，而是偏复用和决策准备的容器。

这三类容器的定位应明确区分：

- 正式分类容器：强调稳定归档
- 未明确来源容器：强调允许保存、后续再补充
- 收藏夹容器：强调复用与回看，不等于饮食分析输入

#### 4.4.2 支持来源

当前支持三类来源：

- `chat_message`
- `estimate_api`
- `manual`

#### 4.4.3 保存规则

- 只有用户主动保存时，才创建 Food Log。
- `chat_message` 来源必须带 `session_id`。
- `chat_message` 只能保存结构化估算结果。
- 文本回复与推荐回复不能直接保存到 Food Log。
- `/estimate` 成功响应也不会自动保存，仍需显式调用保存接口。

下一阶段补充规则：

- Chat 中的可保存菜品卡片，必须在保存前具备明确的分类标签与归档目标。
- 保存动作不应只保存“标题 + 热量结果”，还应保存该卡片对应的目录对象信息或私有对象信息。
- 若一张卡片代表套餐中的子项，则保存时应保留其所属套餐关系与 `item_role`，避免所有子项被错误归入同一个平铺容器。
- 若卡片未命中公共目录但用户选择保存，则应先落到该用户的私有商品对象，再进入对应记录容器。
- 不允许出现“卡片显示正确，但保存后归到错误分类容器”的情况。
- 对于可估算但来源不明确的菜品，不应一律禁止保存；此类对象允许先进入“未明确来源容器”，后续再补分类。
- 收藏动作与保存动作应区分：同一菜品可以被收藏，但不一定立即进入正式分析链路。

#### 4.4.4 幂等与去重

- Food Log 支持 `idempotencyKey`。
- 数据库对 `(user_id, idempotency_key)` 建唯一索引。
- Chat 保存默认使用消息相关 key。
- 独立 estimate 保存默认使用 `estimate_api:{clientRequestId}`。
- 如果命中已删除记录，系统会先恢复再更新，而不是新建重复条目。

#### 4.4.5 列表与检索

后端接口：

- `GET /food-logs`
- `GET /food-logs/{food_log_id}`
- `POST /food-logs`
- `POST /food-logs/from-estimate`
- `PATCH /food-logs/{food_log_id}`
- `DELETE /food-logs/{food_log_id}`
- `POST /food-logs/{food_log_id}/restore`

当前列表过滤能力：

- `sessionId`
- `dateFrom`
- `dateTo`
- `query`
- `sort`
- `limit`

当前排序能力：

- `created_desc`
- `created_asc`

下一阶段的方向补充：

- Food Log 的筛选维度应优先补齐“品类 / 品牌 / 商品来源范围（公共或私有）”。
- 搜索算法会继续增强，但优先级低于“先把记录对象结构化”。
- 没有稳定目录层级之前，不应过早把主要精力投入到复杂搜索排序模型。
- Food Log 的容器组织方式应与分层商品目录保持一致，至少能按“品类 / 品牌 / 品牌商品或私有商品”归档。
- 聊天卡片上的分类标签必须与 Food Log 容器字段一一对应，避免展示层和保存层使用不同对象语义。
- Food Log / Explorer 层应额外提供“未明确来源容器”和“收藏夹容器”的独立查看入口。

当前状态判断：

- “优化 Food Log 的检索与筛选能力”最轻量级部分已完成。
- 现阶段已从仅按 meal 过滤，升级为支持关键词查询与排序。
- 更复杂的多维筛选仍未进入当前实现。

#### 4.4.6 编辑与生命周期

- 删除为软删除。
- 支持恢复已删除条目。
- 前端允许编辑食材克重，并按比例重算热量和三大营养素展示。
- 列表默认只返回 `active` 条目。

#### 4.4.7 图片与元数据

Food Log 当前已支持：

- `image`
- `imageSource`
- `imageLicense`

当前行为：

- 若有图片但未传来源，后端会给默认来源值。
- 若有图片但未传授权，后端默认 `user_owned`。
- 前端无图时展示占位图。
- 详情页展示来源与授权信息。

#### 4.4.8 标准菜品图片自动补齐（当前实现）

当前目标：

- 为高置信命中标准菜品的 Food Log 自动补齐统一、稳定的官方菜品图。
- 提升 Food Log 列表识别效率与整体产品质感。
- 建设可持续维护的“标准菜品视觉资产库”，而不是把图片能力绑定在单条 Food Log 上。

核心策略：

- 图片生成、存储、审核与复用的粒度是 `standard_dish`，不是 `food_log`。
- 每个标准菜品只维护一张官方默认图；同一道标准菜品命中的所有 Food Log 复用同一张图。
- AI 生成按统一任务编排链路触发；审核通过后全局复用，除非人工重生成，否则不重复生成相同官方图。
- AI 生成结果不直接面向用户展示，必须先进入后台审核队列。

触发与复用逻辑：

- 当 Food Log 保存过程中，高置信命中标准菜品时，系统为该记录绑定 `standard_dish_id`。
- 若 `standard_dish.image_url` 为空，则异步触发 `image_generation_job`。
- 若 `standard_dish.image_status = approved`，则 Food Log 可直接复用该标准菜品图片。
- 若未命中标准菜品、命中置信度不足，或图片仍处于 `pending` / `null` 状态，则不阻塞保存，前端继续展示占位图。
- 若用户当前提问本身只得到低置信标准菜品候选，Chat 层应优先发起澄清追问，避免直接产出模糊菜品卡片并把不稳定条目写入 Food Log 或标准菜品资产链路。

统一生成规范：

- 所有标准菜品图必须使用统一的 Prompt Template 生成，禁止不同菜品使用分散、风格不一致的临时 prompt。
- 用户侧输入应保持极简：普通用户或上游业务层只需要提供菜名 / 标准菜品名，不应要求用户手写详细出图 prompt。
- 系统已在图片生成链路内部完成 Prompt 扩写：基于标准菜品名自动补齐构图、容器、视角、摆盘、背景与光线等描述，再与统一风格模板拼装为最终 prompt。
- Prompt 扩写应优先采用可控、可回归的规则模板或受控标签映射，而不是把是否能生成好图完全交给用户输入质量。
- 当系统能识别菜型时，应按菜型补齐细节；例如汤面类补齐“碗装 / 可见汤面 / 热气感”，盖饭类补齐“米饭打底 / 主菜覆盖”，披萨类补齐“圆形薄饼 / 融化芝士”，甜品类补齐“精致摆盘 / 柔和甜品光线”。
- 当菜型识别不足或无法稳定分类时，系统会回退到通用美食摄影扩写模板，而不是直接使用过短的裸菜名 prompt。
- 当前推荐模板要求包括：`菜品主体单一且居中`、`约 45 度轻微俯视角`、`米白/浅灰干净背景`、`柔和自然光`、`真实但略理想化`、`无人物/手/明显餐具干扰`、`无文字/水印/logo`、`适合作为产品卡片封面`。
- 统一 prompt 的目标是保证视觉资产库风格稳定，避免同类菜品出现“照片感、插画感、背景复杂度”明显不一致的问题。

审核与入库流程：

1. AI 根据统一模板生成标准菜品候选图。
2. 候选图进入后台审核队列，并写入候选记录或将标准菜品标记为 `pending`。
3. Admin 页面展示待审核列表，支持查看、通过、拒绝与重新生成。
4. 审核通过后，将图片标记为该标准菜品的官方图，并写回 `standard_dish.image_url` 与 `standard_dish.image_status = approved`。
5. 审核拒绝后，候选图保留审计状态或被丢弃，并允许重新发起生成。

当前落地结构：

- `standard_dishes`
- `dish_images`
- `image_generation_jobs`
- `dish_image_admin_events`

治理要求：

- 标准菜品官方图属于受控产品资产，应继续保留来源与授权审计字段，沿用现有 `image / imageSource / imageLicense` 治理思路。
- Admin 审核通过前，候选图不得直接出现在用户侧 Food Log 列表或详情中。
- Admin 审核页面属于内部运营工具，只对管理员开放，不进入普通用户的信息架构、主导航或常规设置入口。
- 图片补齐是展示增强层，不改变估算结果，不参与识别主链路，也不影响 Food Log 主保存成功与否。

当前阶段不做事项：

- 不做开放式网络搜图或抓图后直接入库。
- 不允许用户上传图片参与标准菜品图建设。
- 不把图片作为菜品识别入口。
- 不承诺所有自由文本菜名都能自动补齐图片。

### 4.6 Insights

后端接口：

- `POST /api/insights/analyze`
- `GET /api/insights/history`
- `GET /api/insights/basket`
- `PUT /api/insights/basket`

#### 4.6.1 当前模型

Insights 当前应采用“左侧实时聚合 + 右侧 AI 快照”的双轨模型，而不是整页统一沿用历史快照。

核心原则：

- 左侧基础数据区与右侧 AI 分析区是两个独立状态源，不能绑定为同一次刷新动作。
- 左侧基础数据区展示的是“当前已选菜品”的实时聚合结果，不依赖用户点击“生成 AI 分析 / 重新分析”。
- 右侧 AI 分析区展示的是显式生成的 snapshot，只有在用户主动触发分析后才更新。
- Food Log、分析篮子或当前日期范围内的已选菜品发生变化时，左侧应立即同步；右侧继续保留上一次 AI 快照，直到用户重新分析。
- 历史结果优先只适用于右侧 AI 分析区，不适用于左侧基础数据区。

#### 4.6.2 分析维度

当前支持：

- `day`
- `week`

分析输入：

- `mode`
- `dateRange`
- `selectedLogIds`（可选）
- `cacheKey`（前端可传）

分析输出：

- `aggregation`：总热量、蛋白、碳水、脂肪、比例、条目数
- `entries`：参与分析的简要条目
- `ai`：`summary / risks / actions`

补充产品语义：

- `aggregation` 属于可由当前已选菜品直接计算出的基础数据，可实时刷新。
- `ai` 属于解释性产物，必须通过显式分析动作生成，不做隐式重跑。

#### 4.6.2A 分析准入规则（下一阶段补充）

详细数据模型规格见：`docs/CONTAINER_DATA_MODEL_SPEC.md`

随着保存容器变多，Insights 的准入规则必须更严格。

核心原则：

- 只有**最底层的菜品 / 商品对象**才能进入营养分析。
- 容器本身不能直接进入分析。
- 收藏夹不能直接整体进入分析。
- 未明确来源容器也不能整体进入分析。

允许进入分析的对象应满足：

1. 它对应一个明确的最底层可食用对象
2. 它有稳定的营养快照
3. 它不是仅用于分组展示的上层容器

这意味着：

- 一个品牌容器不能直接加入分析篮子
- 一个收藏夹不能直接加入分析篮子
- 一个“未明确来源”容器不能直接加入分析篮子
- 只有容器内部的具体菜品项才能被加入分析篮子

对“未明确来源容器”中的对象，允许进入分析，但前提是该对象已经至少稳定到“单个最底层菜品项”层级，而不是一条还未拆清的模糊集合。

#### 4.6.3 历史快照与唯一性

持久化层当前规则：

- `insights_analysis` 以 `user_id + mode + date_start + date_end` 唯一。
- 同一时间范围下重新分析，会覆盖旧结果。
- `selectedLogIds` 会被保存，但不会形成同一时间范围下的多版本并存。

这意味着：

- 当前版本的产品语义是“每个时间范围保留最新分析快照”。
- 它不是“同一天/同一周可保存多套筛选版本”的历史版本系统。

#### 4.6.4 左右分区职责与更新规则

左侧基础数据区包括但不限于：

- 总热量
- 宏量营养素分布
- 已选菜品列表
- 基于当前已选菜品可直接得出的计数、进度和汇总指标

右侧 AI 分析区包括但不限于：

- AI 饮食建议
- 综合评估
- 风险提示
- 行动建议
- “生成 AI 分析 / 重新分析”触发入口

更新规则：

1. 只要当前日期范围内有菜品被添加、删除、替换、编辑，或分析篮子选择发生变化，左侧基础数据区就应立即更新。
2. 左侧更新不应要求用户点击“重新分析”，也不应等待历史快照返回。
3. 右侧 AI 分析区继续维持显式触发语义，仅在用户点击“生成 AI 分析 / 重新分析”后更新。
4. 当左侧当前数据与右侧 AI 快照的输入不一致时，应明确提示“右侧 AI 结果可能已过期/基于历史快照”，但不能阻塞左侧刷新。
5. 左侧和右侧可以同时展示不同时间点的数据语义：左侧是当前选择，右侧是最近一次分析快照；这种差异必须被产品文案清楚表达。

#### 4.6.5 页面加载逻辑

当前前端行为：

1. 先读取当前日期范围内的已选菜品/分析篮子，用于计算左侧基础数据区
2. 左侧基础数据区优先展示当前实时聚合结果
3. 同时读取 `GET /api/insights/history`，用于恢复右侧最近一次 AI 快照
4. 若当前时间范围已有历史结果，则右侧展示该快照
5. 若没有历史结果，则右侧展示空状态并提供“生成 AI 分析”入口
6. 分析成功后，仅更新右侧 AI 结果快照，并将当次输入保存为该时间范围的最新快照
7. 后续若菜品继续变化，左侧继续实时更新；右侧保持最近一次分析结果，并显示时效提醒

#### 4.6.6 分析篮子同步

当前实现已支持分析篮子后端同步：

- 前端本地仍保留 basket 状态，保证交互即时性
- 同时通过 `/api/insights/basket` 做账号级同步
- 换设备或重新登录后，可恢复已同步的 basket 项

当前边界：

- basket 是“待分析条目选择状态”，不是正式分析历史
- 它没有版本比较或多人协作语义

补充要求：

- basket 变化应首先驱动左侧基础数据区刷新，因为它直接改变“当前已选菜品”的集合。
- basket 不应自动触发新的 AI 分析任务；是否更新右侧 AI 文本，仍由用户显式决定。
- basket 的加入对象应限制为最底层菜品项；若用户当前点击的是上层容器，应先展开并选择具体菜品。

#### 4.6.7 周分析展示状态

当前周分析已具备日/周双模式、历史快照、AI 文本输出与趋势上下文注入能力。

现阶段已经支持：

- 基础周视图
- 基础聚合结果
- AI 周期总结
- 按天序列、前后半周变化、波动等级与周期判断

仍待后续增强的部分包括：

- 更长周期对比（如 30 天）
- 同一时间范围的多版本比较
- 更强的结构化趋势评分与长期跟踪表达

## 20. 决策助手升级任务总览

以下任务用于把 FoodPilot 从“以聊天为主入口的营养助手”，升级为“面向点单场景的 AI 饮食决策助手”。这些任务不是并行等权堆叠，而是应按依赖顺序推进，优先完成最小闭环，再逐步补强输入、品牌精度和个体化能力。

### 20.1 升级主链路

目标主链路应调整为：

`输入商品信息 -> 解析商品结构 -> 估算营养与置信度 -> 输出推荐等级/风险/建议 -> 用户决定是否保存或继续比较`

### 20.2 本轮规划的设计原则

1. 优先复用现有 `Chat / Estimate / Profile / Food Log / Insights` 能力，不另起一套完全独立系统。
2. 先完成“标题/描述输入 -> 决策输出”的可用闭环，再扩展截图、链接、分享等入口。
3. 所有新能力都必须有明确结构化输出契约，而不是只依赖自由文本回答。
4. 商品数据底座优先采用“品类 / 品牌 / 品牌商品”的分层目录，而不是继续堆平铺 alias。
5. 对未收录长尾商品，优先支持用户私有补充和私有收藏夹，而不是要求公共库先全覆盖。
6. 搜索与召回需要做，但当前优先级低于目录结构、商品映射和私有补充能力。
7. 优先支持高频品牌、高频品类和高频用户目标，不追求一次性覆盖所有平台商品；品牌侧优先建设有体量的连锁品牌。
8. 每个阶段都要有可验证的验收标准，避免只完成概念设计而没有上线价值。

### 20.3 任务优先级分层

| 优先级 | 目标 | 对应任务 |
| --- | --- | --- |
| P0 | 完成“场景输入到决策输出”最小闭环 | Task 1 ~ Task 5 |
| P1 | 提升品牌感知、配置识别和截图能力 | Task 6 ~ Task 8 |
| P2 | 提升长期使用价值和多入口扩展能力 | Task 9 ~ Task 10 |

7. 增加套餐拆分逻辑，能把“汉堡+薯条+可乐”“双人便当套餐”等输入拆成多个组成项。
8. 设计“未录入商品”的处理分支：
   - 若命中品类和品牌但未命中商品，允许用户创建私有商品
   - 若仅命中品类，允许用户在该品类下补充品牌和商品
   - 私有商品默认只进入用户自己的收藏夹
9. 设计“来源不明确但可估算”的处理分支：
   - 允许保存到未明确来源容器
   - 允许后续人工补品牌、补做法、补来源
   - 不要求一开始就命中正式品牌商品
10. 对解析结果输出置信度、缺失字段和 `analysis_eligible`，供上层判断是否需要追问澄清、允许保存或允许进入分析。
11. 为高频真实样本建立回归集，避免每次加规则都破坏既有识别。
12. 为“无品牌 / 自制菜品”建立专门的澄清分支，确保系统会主动追问影响热量差异的关键因素，而不是只在品牌命中失败后停止。

### 目标成果

- FoodPilot 有一个独立的“商品理解层 + 分层商品目录层”，而不是直接把原始文本扔给估算逻辑。
- 系统能区分品类、品牌、品牌商品、规格和附加项，具备品牌感知估算的基础。
- 复杂套餐可以拆成结构化组成项，而不是只按整体标题粗估。
- 未被公共目录收录的长尾商品，也有明确的用户私有补充路径。
- 无品牌和自制菜品也有可用估算路径，不会因为缺少品牌而直接失效。
- 来源不明确但已可估算的菜品，有专门的保存容器，不需要在“禁止保存”和“强行正式归档”之间二选一。

### 验收标准

- 高频品牌和高频品类样本能稳定输出标准化结构，并挂靠到明确目录层级。
- 套餐输入至少能正确拆出主商品、饮品、配餐三个层次中的两个以上。
- 当商品未命中公共目录时，存在清晰的用户私有补录流程。
- 解析低置信时能明确暴露缺失点，而不是伪装成高确定结果。
- 无品牌 / 自制菜品输入时，系统能主动发起具体澄清问题，并在补充后继续给出可解释估算。
- 来源不明确对象可以进入未明确来源容器，但只有满足最底层菜品条件时才可被标记为 `analysis_eligible`。

### Task 4：实现品牌感知估算层 V1

### 目标

从“通用菜名估算”升级为“品牌模板 + 品类模板 + 配置修正”的估算体系，先解决同类商品跨品牌差异过大的问题。

### 需要完成的步骤

1. 确定首批优先覆盖的品牌和品类：
   - 饮品品牌：优先喜茶、霸王茶姬、瑞幸、库迪这类有体量的连锁品牌
   - 快餐品牌：优先麦当劳、肯德基、塔斯汀这类有体量的连锁品牌
   - 泛连锁便当/轻食模板
2. 为每个高频品牌建立基础模板；首批模板不优先覆盖零散单店，而优先覆盖标准化程度高、复用价值高的连锁品牌：
   - 基础热量区间
   - 宏量营养素估算
   - 主要热量来源
   - 适用规格
3. 建立配置修正规则：
   - 糖度调整
   - 小料增加
   - 套餐饮品替换
   - 配餐升级
   - 份量变化
4. 当品牌模板不存在时，回退到品类模板，再回退到通用模板，并显式降低置信度。
5. 输出估算解释摘要，让用户知道结果依据的是品牌模板、品类模板还是泛化回退。
6. 为品牌模板建立版本化维护方式，便于后续继续扩容和人工校正。
7. 对于无品牌或自制菜品，增加“家常做法 / 通用做法”估算路径，并与澄清问题联动，例如根据甜咸口、用油量、是否勾芡、是否带浇头等信息调整估算。

### 目标成果

- 相同品类在不同品牌下可以给出明显更合理的差异化估算。
- 系统具备“模板 + 修正”的稳定扩展方式，而不是每次只改 prompt。
- 决策层可以直接复用估算输出和置信度。
- 对无品牌、自制和品牌未知场景，系统也能通过品类模板和追问机制给出可用估算。

### 验收标准

- 首批高频品牌的核心商品能走品牌模板而不是泛化模板。
- 糖度、加料、规格变化会影响估算结果，而不是只影响文案。
- 返回结果中可区分高置信品牌估算和低置信泛化估算。
- 品牌未知或自制场景不会被直接拒绝，而会在澄清后进入品类模板或通用模板估算。

### Task 5：实现个体化决策层与建议层 V1

### 目标

让系统从“告诉你大概多少热量”，升级为“告诉你对你是否值得点、风险在哪、如果还想点该怎么调”，并且在不打破用户整体饮食节奏的前提下，保留合理的快乐空间。

### 需要完成的步骤

1. 将 `Profile` 中已有字段整理为决策输入上下文：
   - 目标
   - 热量目标
   - 饮食风格
   - 过敏原
   - 忌口 / 限制
2. 设计推荐等级规则：
   - 推荐
   - 谨慎
   - 暂不推荐
   - 规则解释必须避免道德化表达，不能把“偶尔想吃”默认表述为失败或失控
3. 设计风险标签体系，优先覆盖：
   - 高糖
   - 高油
   - 高碳
   - 蛋白不足
   - 份量过大
   - 过敏风险
4. 设计“仍然想点”的建议生成逻辑，至少包括：
   - 同商品改配
   - 套餐替换
   - 更适合当前目标的同类替代
   - 保留快乐主体、调整次要项的折中方案
5. 增加“节奏友好说明”字段，明确告诉用户：
   - 当前选择是否属于可接受范围内的享受
   - 是否需要当天稍作平衡
   - 是否无需因为一次选择而过度补偿
6. 将建议输出与估算结果、置信度、商品结构绑定，避免建议脱离上下文。
7. 在决策层明确用户未建档时的默认策略，避免系统因为缺少 Profile 而不可用。
8. 为不同目标用户建立代表性测试样本，例如减脂、控糖、增肌三类，并增加“快乐型选择”样本，验证系统不会一味劝退。

### 目标成果

- FoodPilot 可以输出明确的“是否推荐”结论，而不是只有营养数字。
- 用户即使不放弃商品，也能得到可执行的改配建议。
- 系统能表达“这次可以吃，但怎么吃更稳妥”，而不是只能表达“不要吃”。
- 个体目标开始真正影响系统输出，体现决策系统价值。

### 验收标准

- 同一商品面对不同 Profile 时，推荐等级和建议能发生合理变化。
- 风险标签不是装饰字段，而是与估算和规则一致。
- 对存在享受价值但总体可控的商品，系统能给出“允许但有边界”的建议，而不是机械劝退。
- 至少能稳定覆盖减脂、控糖、增肌三类典型场景。

### Task 6：上线决策工作台前端 V1

### 目标

把前端从“聊天消息流展示结果”，升级为更适合点单决策的工作台式界面，让输入、结果、理由和建议都可快速扫读；同时让聊天中的菜品分析卡片与后续保存容器的分类逻辑保持一致。

### 需要完成的步骤

1. 设计新的结果卡片层级，至少包含：
   - 商品主结论
   - 分类标签区
   - 估算摘要
   - 风险标签
   - 推荐等级
   - 调整建议
   - 替代建议
2. 明确澄清态、低置信态和成功态的 UI 区别，不能都长得一样。
3. 为套餐结果设计分组展示，区分主商品、饮品、配餐和整体合计。
4. 在卡片上显式展示最关键的归档标签，至少包括：
   - 当前所属品类
   - 当前所属品牌
   - 当前命中的商品或“用户私有商品”
   - 当前 `item_role`
5. 在卡片上明确展示当前保存去向与分析资格，例如：
   - 将保存到正式分类容器
   - 将保存到未明确来源容器
   - 已加入收藏夹
   - 当前不可直接加入分析
6. 为用户提供二次操作入口：
   - 修改标题/描述重新分析
   - 调整目标后重新判断
   - 保存到 Food Log
   - 加入收藏夹
   - 继续比较其他商品
7. 保存前若标签不完整，但已达到可估算和可保存条件，应允许进入未明确来源容器，而不是一律拦截。
8. 对移动端进行单独适配，保证真实点单场景下可在手机上快速使用。
9. 优化文案，把“估算工具语气”升级为“帮助做选择的决策语气”。

### 目标成果

- FoodPilot 前端表现形式从聊天问答明显升级为决策辅助界面。
- 用户可以快速扫读“值不值得点”和“如何改”。
- 用户在聊天界面就能看出该卡片保存后会归入哪个分类容器。
- 用户也能看出该卡片能否加入营养分析，避免把保存和分析混为一谈。
- 现有 Workspace 成为升级入口，而不是需要新做一个完全独立站点。

### 验收标准

- 用户在单屏内即可看到输入、结论、风险和建议四个核心区块。
- 可保存卡片会明确展示分类标签，且保存后归档结果与卡片标签一致。
- 收藏、保存、加入分析三种动作的准入条件在 UI 上清晰分离。
- 移动端使用时不需要横向滚动，也不会出现信息被长文本淹没的问题。
- 保存与重新分析入口清晰可见，不需要回到聊天上下文里找。

### Task 7：接入截图识别与 OCR 入口

### 目标

在标题/描述输入稳定后，增加真实平台截图解析能力，降低用户复制粘贴成本。

### 需要完成的步骤

1. 设计截图上传入口与上传限制：
   - 文件大小
   - 支持格式
   - 隐私提醒
2. 明确 OCR / 多模态识别链路输出的中间结构：
   - 识别文本
   - 候选商品标题
   - 候选规格
   - 候选品牌
3. 增加截图解析后的人工确认步骤，避免 OCR 错误直接进入决策。
4. 建立截图识别失败、文字过密、页面遮挡、促销图干扰等异常分支。
5. 将截图识别结果复用到 Task 3 的商品理解层，而不是单独维护另一套解析逻辑。
6. 为典型平台截图建立样本集和回归验证集。

### 目标成果

- 用户可以直接上传商品截图，而不必手动整理文案。
- 截图能力成为输入层增强，不会破坏既有标题/描述链路。
- 多模态入口和文本入口在后端统一汇聚到同一商品理解层。

### 验收标准

- 截图上传后至少能提取出商品主体文本并进入确认流程。
- OCR 低质量时会要求用户确认，而不是直接给出高置信结论。
- 文本入口和截图入口最终输出同一种标准化结构。

### Task 8：补强品牌模板、配置规则与置信度机制

### 目标

在 V1 可用闭环基础上，提高结果稳定性、可解释性和用户信任度。

### 需要完成的步骤

1. 扩充高频品牌模板覆盖面，优先围绕真实用户高频输入扩容。
2. 补强配置识别规则，覆盖：
   - 冰量
   - 糖度
   - 奶基底
   - 加料
   - 配餐替换
   - 双人/多人套餐份量变化
3. 设计统一置信度机制，至少包含：
   - 高置信：品牌模板命中 + 规格明确
   - 中置信：品类模板命中 + 局部配置缺失
   - 低置信：仅通用模板回退或套餐信息不足
4. 在前端明确展示置信度与原因说明。
5. 为“需要补问”的场景建立结构化追问策略，而不是通用“请补充更多信息”。
6. 建立模板和规则的维护清单与版本变更记录。

### 目标成果

- 结果不只是“更准一点”，而是“更可解释、更可信”。
- 系统能明确告诉用户当前结论基于什么程度的信息完整性。
- 运营和研发后续可以持续维护模板而不失控。

### 验收标准

- 用户能直观看到结果是否属于高、中、低置信。
- 不同置信度会触发不同 UI 和不同建议策略。
- 扩充模板不会显著破坏已有回归样本。

### Task 9：打通比较、保存与长期使用链路

### 目标

让 FoodPilot 不只处理单次商品判断，而是开始支持“两个商品哪个更适合我”和“本周我总是怎么选”的长期价值。

### 需要完成的步骤

1. 设计两商品比较输入和结果结构，支持：
   - 商品 A / 商品 B 对比
   - 同类替代对比
   - 套餐与单品对比
2. 明确比较维度：
   - 热量
   - 三大营养素
   - 风险标签
   - 推荐等级
   - 更优改配路径
3. 优化保存逻辑，让决策卡片可选择性沉淀到 Food Log 或后续专门的 decision history。
4. 让保存结果尽量挂靠到“品类 -> 品牌 -> 品牌商品”层级；未收录商品则挂靠到用户私有收藏夹对象或未明确来源容器。
5. 沿用并收敛现有收藏夹功能，使其成为独立容器，而不是混入正式分析链路。
6. 保存时同步校验卡片分类标签与实际落库容器是否一致，避免出现错误归档。
7. 将历史选择与 Profile、Insights 形成弱连接，用于后续分析用户常见风险和高频选择模式。
8. 在 Insights 或其他后续页面中预留“近期点单决策回顾”能力。

### 目标成果

- FoodPilot 开始具备“比较和复盘”能力，而不是一次性问答工具。
- 用户可以把点单判断与后续饮食分析连接起来。
- Food Log 与用户收藏夹开始共享稳定的商品对象层，而不是只共享自由文本。
- 聊天卡片、保存动作和最终容器三者使用同一套分类对象语义。
- 未明确来源容器、收藏夹容器与正式分析链路之间边界清晰。
- 为长期 Agent 化铺设上下文基础。

### 验收标准

- 两商品比较结果有明确优先结论和理由。
- 用户可以回看关键决策记录，而不是只看营养流水。
- 已保存记录可以区分公共商品与用户私有商品来源。
- 卡片标签、落库对象和容器归档结果一致，不存在跨层级错放。
- Insights 只接收最底层菜品对象，不接收上层容器对象。
- 历史沉淀不破坏现有 Food Log 语义。

### Task 10：建立数据沉淀、评估与运营机制

### 目标

让该方向不仅能做出 demo，还能持续提高结果质量和覆盖面。

### 需要完成的步骤

1. 建立真实输入样本池，按品牌、品类、目标、输入方式分类沉淀。
2. 建立商品理解评估集，衡量：
   - 品牌识别准确率
   - 规格识别准确率
   - 套餐拆分准确率
3. 建立决策结果评估集，衡量：
   - 推荐等级一致性
   - 风险标签合理性
   - 替代建议可执行性
4. 为品牌模板和修正规则建立版本记录与回滚机制。
5. 建立数据缺口清单，持续发现哪些品牌、品类、场景最值得优先补齐。
6. 搜索与召回能力在该阶段进入专项优化，但前提是目录层级和私有补充数据已经稳定。
7. 明确内部运营职责边界：
   - 谁维护模板
   - 谁审核截图样本
   - 谁更新评估集

### 目标成果

- FoodPilot 具备持续迭代的质量抓手，而不是靠主观感觉调 prompt。
- 品牌扩容、规则扩容、截图扩容都有统一评估框架。
- 公共目录、用户私有商品和后续搜索优化之间有清晰边界。
- 后续无论做浏览器插件、小程序还是平台合作，都有稳定的数据底座。

### 验收标准

- 至少存在一套可持续维护的评估集和质量基线。
- 每次规则或模板更新后都能判断是否真实提升了系统质量。
- 搜索优化建立在稳定目录和稳定样本之上，而不是脱离商品对象层单独调参。
- 新品牌或新品类的接入流程有明确 SOP，而不是一次性人工 patch。

## 22. 建议执行顺序

建议按以下顺序推进，避免同时改太多层导致链路失控：

1. 先完成 Task 1，统一输出契约。
2. 再完成 Task 2 和 Task 3，建立稳定输入层与商品理解层。
3. 接着完成 Task 4 和 Task 5，形成真正的决策结果。
4. 然后完成 Task 6，把前端体验升级为决策工作台。
5. 在 P0 闭环稳定后，再做 Task 7 和 Task 8，补强截图和品牌精度。
6. 最后推进 Task 9 和 Task 10，沉淀长期价值和运营机制。

## 23. P0 阶段定义

只有以下结果都成立，才能认为“FoodPilot 已从分析工具升级为决策助手的最小版本”：

1. 用户可直接输入商品标题或描述，不必组织聊天式问句。
2. 系统已建立最小可用的“品类 -> 品牌 -> 品牌商品”分层目录。
3. 系统能输出统一决策卡片，而不只是自由文本或热量数字。
4. 结果包含估算、推荐等级、风险提示、个体适配说明和至少一条可执行建议。
5. 高频品类至少覆盖奶茶、咖啡、汉堡/炸鸡、便当四类。
6. 对未收录商品，用户有可用的私有补录与收藏路径。
7. 低置信输入会进入澄清或降级提示，不会伪装成高确定答案。
8. 前端已经具备适合真实点单场景的移动端可用界面。
