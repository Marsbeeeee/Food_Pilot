## 1. 当前完成度评估（代码基线）

### 1.1 模块完成度

- `Assistant（聊天主入口）`：高（核心可用）
- `Food Log（保存与复用）`：高（核心可用）
- `Profile（个性化约束）`：高（核心可用）
- `Auth / Session（账号与会话）`：高（核心可用）
- `Insights（日/周分析闭环）`：中低（有 UI 雏形，未形成完整产品闭环）
- `内容数据层（菜品图片与本地化营养知识）`：低（尚未形成可持续数据源）
- `工程化（前端测试、配置、可观测性）`：中低（可运行，规模化能力不足）

### 1.2 当前已落地能力（事实基线）

1. 用户可注册/登录，支持会话恢复与删除账号。
2. 聊天支持创建会话、续聊、重命名、删除会话。
3. Assistant 能处理两条核心路径：
   - 推荐吃什么（`meal_recommendation`）
   - 这餐大概多少热量/营养（`meal_estimate`）
4. 估算结果支持显式保存到 `Food Log`，且支持撤销保存。
5. Food Log 支持列表、详情、编辑、删除、恢复、跳回来源聊天。
6. Profile 已接入推荐链路，包含过敏原硬约束校验。

### 1.3 当前主要缺口

1. `Insights` 仍是“前端本地聚合 + UI 展示”，不是完整业务模块。
2. 缺少“按周分析、趋势对比、历史分析记录”闭环。
3. Insights 的 AI 分析默认仍依赖前端 fallback 文案，未稳定接入后端分析能力。
4. 前端 API 地址仍硬编码，环境切换与部署能力不足。
5. 前端缺少自动化测试，回归风险主要靠手工验证。
6. Food Log 条目缺少稳定的真实图片来源与回填策略，影响可读性与复用体验。
7. 当前估算知识对中文饮食（家常菜、地方菜、中餐份量表达）适配不足，结果偏泛化。

## 2. 产品定位（当前版本冻结）

Food Pilot 是一个以对话为入口的营养助理。
主路径保持为：

`Assistant -> Save to Food Log -> 进入 Insights 做日/周分析 -> 获取调整建议`

### 2.1 当前范围内做什么

- 做：推荐、估算、保存复用、资料约束、基础分析入口
- 不做：医疗诊断、临床营养方案、全量饮食流水账

### 2.2 产品边界（继续保留）

- `Food Log` 是“用户主动保存的高价值结果”，不是自动记录全部饮食。
- `/estimate` 是后端能力接口，不是产品主入口。
- 过敏原约束是硬规则，不是提示性文案。

## 3. 下一阶段要完善什么（优先级）

## 3.1 P0（必须优先完成）

### P0-1：把 Insights 从“页面”升级为“闭环能力”

目标：让 Insights 成为可依赖的核心模块，而不是展示壳层。

必须交付：

1. 支持按天分析和按周分析两种模式。
2. 支持总热量、蛋白/碳水/脂肪汇总及结构判断。
3. 支持对“所选 Food Log 条目”生成 AI 建议（后端接口，不依赖前端 fallback）。
4. 分析结果可复查（至少保存最近 N 次分析记录，包含日期和条目列表）。
5. 分析过程明确显示数据来源（来自哪些 Food Log 条目）。

验收标准：

- 用户可在 1 分钟内完成“选条目 -> 出分析 -> 看建议”。
- 同一日期重复进入时，能看到已选条目或历史分析记录。
- 不依赖前端 mock/fallback 才能生成结果。

### P0-2：稳定 Assistant 意图分流质量

目标：降低误判，提高结果一致性。

必须交付：

1. 路由规则扩展为“规则 + 回归样例”，覆盖中文高频问法。
2. 对边界输入增加澄清策略（不确定时询问用户偏向推荐还是估算）。
3. 明确 `meal_recommendation` 与 `meal_estimate` 的输出契约，前后端一致。

验收标准：

- 核心回归样例准确率达到目标阈值（建议 >= 90%）。
- 前端不再依赖“猜测字段”渲染消息类型。

### P0-3：补齐工程化底线

必须交付：

1. 前端 API Base URL 环境化（开发/测试/生产可切换）。
2. 前端关键流程自动化测试（至少覆盖登录、发消息、保存 Food Log、进入 Insights）。
3. 基础错误可观测（关键接口失败可追踪）。

验收标准：

- 新环境无需改源码即可启动。
- 主流程每次发布前可自动回归。

### P0-4：接入 Food Log 真实图片数据源

目标：让 Food Log 的菜品记录具备稳定、可信、可展示的图片来源。

必须交付：

1. 定义图片来源策略（优先级）：用户上传 > 合规第三方图库/API > 系统占位图。
2. 为 `Food Log` 增加 `imageSource` 与 `imageLicense` 元信息，避免来源不明。
3. 建立图片回填流程：新保存条目自动尝试匹配图片，失败则进入待补队列。
4. 增加“无图兜底体验”（统一占位图 + 可手动补图入口）。

验收标准：

- Top 高频菜品中，目标覆盖率达到设定阈值（例如 >= 80%）。
- 任一图片可追溯来源与授权信息。
- 无图场景不影响核心使用路径。

### P0-5：中文饮食估算能力增强（RAG 优先，微调后置）

目标：让模型在中国饮食场景下更准确、更可解释。

必须交付：

1. 建立“中国饮食知识库”RAG（食材、常见菜品、份量换算、烹饪方式、营养基准）。
2. Prompt 增加中餐语境约束（家常菜名、外卖表达、地方叫法、克重/份量语义）。
3. 增加中文估算回归集（按菜系与场景分层：家常菜、快餐、夜宵、地方菜）。
4. 建立数据治理规则：仅接入有授权的数据源，不直接抓取或复用受限 App 私有数据。
5. 微调仅在 RAG 稳定且评估显示瓶颈后启动，避免过早投入。

验收标准：

- 中文菜品估算基准集误差显著下降（按项目指标定义）。
- 输出中可解释“估算依据”（食材、份量、烹饪方式）。
- 数据源合规可审计。

## 3.2 P1（应尽快完成）

1. Food Log 筛选能力增强（按时间、餐次、来源、关键词）。
2. Insights 增加趋势视图（近 7 天/近 30 天）。
3. Profile 与 Insights 深度联动（对比目标热量与实际摄入差异）。
4. 统一中英文文案风格，默认对中文用户全链路中文化。
5. 评估并推进模型微调（以 RAG 评估结果为前置条件）。

## 3.3 P2（可后置）

1. 导出分析报告（图片/PDF）。
2. 更长期的饮食 coaching 模块。
3. 多端适配（移动端深度优化）。

## 5. 功能规范（按模块）

### 5.1 Assistant

必须：

1. 继续只聚焦两类核心意图（推荐、估算）。
2. 推荐结果默认不直接写入 Food Log。
3. 估算结果可一键保存到 Food Log（显式动作）。
4. 推荐链路必须继续执行过敏原硬约束。

### 5.2 Food Log

必须：

1. 保持“主动保存、可复用”的定位。
2. 支持编辑、删除、恢复、跳回来源聊天。
3. 可作为 Insights 的标准输入源。

应增强：

- 更强筛选和检索
- 与 Insights 的跨日期选择体验

### 5.3 Insights

必须：

1. 支持日/周分析。
2. 支持 AI 建议（后端生成）。
3. 支持分析结果留存与复查。
4. 明确“本次分析用了哪些 Food Log 条目”。

### 5.4 Profile

必须：

1. 过敏原约束始终生效。
2. 目标、热量目标、饮食风格要进入推荐与分析上下文。
3. Profile 缺失时需显式提示“个性化不完整”。

### 5.5 Food Data & Knowledge（新增）

必须：

1. 菜品图片有可追溯来源，不使用来源不明图片。
2. 估算优先使用中文饮食知识库 + 结构化营养数据（RAG）。
3. 建立“份量表达标准化”能力（如“一碗”“一份”“一勺” -> 近似克重）。
4. 数据接入遵守授权与合规，不直接复用受限第三方 App 私有数据。

建议策略：

- 先做 RAG 与规则增强，再评估是否微调。
- 微调目标聚焦：中文菜名识别、份量理解、烹饪方式对热量影响。

## 6. 非功能要求（本版新增硬要求）

1. **可配置性**：前端与后端地址均可通过环境配置，不改源码。
2. **可测试性**：主路径必须有自动化回归。
3. **可观测性**：关键失败可定位（接口、AI、鉴权）。
4. **一致性**：消息类型契约、页面文案、保存规则保持统一。
5. **数据合规性**：图片与营养数据来源可审计，授权边界清晰。

---

## 7. 成功标准（执行口径）

满足以下条件视为本阶段成功：

1. 用户能稳定完成“问 -> 保存 -> 分析 -> 调整”的主闭环。
2. Insights 不再依赖前端 fallback 才能生成分析建议。
3. 回归测试可覆盖核心链路并在发布前执行。
4. 配置切换不需要改前端 API 源码。

---

## 8. 暂不纳入本阶段

1. 医疗结论、诊断、治疗建议。
2. 自动记录所有饮食行为的完整日记产品。
3. 复杂长期教练系统（multi-week coaching）作为当前主交付。
