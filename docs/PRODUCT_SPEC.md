## 文档信息

- 文档状态：已按当前仓库实现重整
- 更新时间：2026-03-25
- 事实来源：以后端路由、服务层、前端页面、数据库初始化脚本、测试基线为准

---

## 1. 产品定位

Food Pilot 是一个以对话为主入口的营养助手。当前版本的核心价值不是“自动记全量饮食流水”，而是：

1. 用户通过 Chat 提问，获得餐食推荐或营养估算。
2. 用户将有价值的结果显式保存到 Food Log。
3. 用户基于保存结果进入 Insights 做日/周分析。
4. Profile 为推荐、估算和安全约束提供个性化上下文。

当前产品更接近：

`对话式营养助手 + 显式保存的结果库 + 基础饮食分析面板`

而不是：

- 自动饮食日记
- 医疗诊断工具
- 完整的多模态拍照识别产品

---

## 2. 当前信息架构

前端当前实际由四个一级视图组成：

- `WORKSPACE`：Chat 主工作区
- `EXPLORER`：Food Log 列表与详情
- `INSIGHTS`：复用 Explorer 内的分析视图
- `PROFILE`：个人档案

主链路为：

`注册/登录 -> Chat 提问 -> 获得推荐/估算 -> 显式保存到 Food Log -> 加入 Insights 分析篮子 -> 生成日/周分析 -> 回到 Chat 或 Profile 继续调整`

## 4. 功能规格

### 4.1 Auth

后端接口：

- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`
- `DELETE /auth/me`

当前约束：

- 注册需要合法邮箱、非空 `displayName`、至少 8 位密码。
- 使用 Bearer Token。
- Token 为后端自签 HMAC JWT，包含 `sub / iat / exp`。
- 前端会将 token 与当前用户信息保存在本地。
- 删除账号后，数据库通过外键级联删除该用户相关 Profile、Chat、Food Log、Insights 数据。

### 4.2 Chat / Assistant

#### 4.2.1 会话能力

后端接口：

- `POST /chat/sessions`
- `GET /chat/sessions`
- `GET /chat/sessions/{session_id}`
- `PATCH /chat/sessions/{session_id}`
- `DELETE /chat/sessions/{session_id}`
- `POST /chat/messages`
- `POST /chat/sessions/{session_id}/messages`

当前行为：

- 支持新建空会话。
- 支持在空会话中直接发首条消息并自动生成会话。
- 首条用户消息会用于更新默认标题。
- 删除会话为永久删除，不做软删除。
- 删除会话后，Food Log 记录仍保留，但关联的 `session_id` 会被清空，前端展示为 `Source Chat Deleted`。

#### 4.2.2 消息类型契约

对外消息类型统一为：

- `text`
- `meal_estimate`
- `meal_recommendation`

兼容规则：

- 数据库内部仍可能存储历史值 `estimate_result`
- Router / Schema 层统一映射为 `meal_estimate`

#### 4.2.3 意图分流

当前实现是“规则分流”，不是 LLM 分类器。

分流结果：

- `meal_estimate`：命中“多少热量 / 多少蛋白质 / 营养怎么样 / 大概多少”等估算关键词
- `meal_recommendation`：命中“推荐 / 替代 / 选哪个 / 训练后吃什么 / 怎么优化”等推荐关键词
- `text`：命中解释、说明、区别、为什么等文本类关键词
- `_clarification`：输入模糊、像问题但又不像具体食物描述时，先返回澄清提问

补充说明：

- 估算优先级高于推荐。
- 解释型追问会优先走 `text`，避免把“为什么推荐这个”误判成新的推荐请求。
- 当用户输入触发了估算意图，但菜品标准化匹配置信度不足时，Chat 应优先进入 `_clarification` 追问，而不是直接生成结构化估算结果。
- 典型场景包括“炒面热量”“盖饭热量”“套餐多少卡”这类标准菜品指向不清、缺少关键修饰词或可能对应多个标准菜的表达。
- 当前规则对高频中文问法已有回归测试覆盖，但复杂表述仍存在误分流风险。

#### 4.2.4 Assistant 输出边界

- Chat 内估算结果不会自动写入 Food Log。
- Chat 内推荐结果不能直接保存到 Food Log。
- 多食物估算时，消息 `payload.estimates[]` 会保留每个子估算块，前端可逐块展示。
- 若标准菜品匹配仍处于低置信状态，Chat 不应直接输出可保存的信息卡片，而应先追问用户补足关键信息，如做法、主配料、甜度、品牌、是否带浇头等。

#### 4.2.5 推荐安全约束

当前实现：

- 当请求携带 `profileId` 时，推荐链路会读取 Profile。
- 如果 Profile 中存在 `allergies`，系统会对推荐结果做关键词级过敏原拦截。
- 命中过敏原时，不返回原推荐，而返回“推荐已拦截”的安全提示。

当前边界：

- 这是关键词级拦截，不是结构化配料理解。

### 4.3 独立 Estimate API（系统能力层）

后端接口：

- `POST /estimate`

#### 4.3.1 产品定位

`/estimate` 是系统级的结构化营养估算能力接口，不作为当前版本的用户主入口。

其主要职责：

- 为 Chat 提供底层估算能力
- 为 Food Log 提供结构化数据来源
- 为未来快速录入、批量估算和外部接入提供能力基础

#### 4.3.2 与 Chat 的关系

- Chat 是用户主入口，负责意图理解、澄清和结果组织。
- `/estimate` 是原子能力层，仅负责单次估算计算。
- Chat 在命中 `meal_estimate` 时，应通过统一估算链路调用 `/estimate` 对应能力，而不是发展独立估算逻辑分支。
- 当菜品标准化匹配置信度不足时，Chat 先负责澄清追问；只有在补足信息、达到可稳定映射标准菜品的条件后，才继续进入结构化估算与后续保存链路。

#### 4.3.3 当前入口策略

- 当前版本不提供独立 Estimate 页面或主按钮入口。
- 所有用户估算行为优先通过 Chat 触发。
- `/estimate` 仅作为内部能力和前端 client 保留。

#### 4.3.4 后续演进方向（非当前版本）

未来可能扩展为：

- Quick Add（无对话快速录入）
- 批量食物估算入口
- 第三方 API / 多端复用能力
- 面向中国用户的标准化食物数据底座

其中“面向中国用户的标准化食物数据底座”下一阶段的产品要求为：

- 继续扩充中国高频菜品、早餐、外卖套餐、饮品、便利店与健身餐覆盖。
- 为知识条目建立稳定 `food_id`、标准名、中文别名、份量提示、营养值来源与更新时间。
- 优先提升“中文真实菜品匹配与估算稳定性”，而不是先追求通用国际食物大全。
- 数据形态允许继续以离线版本化数据集为主；是否迁移为独立表或独立数据库，取决于后续后台运营、批量维护与多来源同步需求。

#### 4.3.5 不做事项（当前阶段）

- 不作为 Chat 的替代入口
- 不提供与 Chat 重叠的推荐或解释能力
- 不在 UI 中形成与 Chat 并列的主功能模块

#### 4.3.6 请求与返回结构

请求支持：

- `query`
- `clientRequestId`
- `profileId`
- `sessionId`

返回结构包含：

- `success`
- `data`
- `error`
- `clientRequestId`
- `foodLogId`
- `saveStatus`

### 4.4 Food Log

#### 4.4.1 产品定位

Food Log 是“显式保存的结果库”，不是自动记录所有饮食行为的流水账。

#### 4.4.2 支持来源

当前支持三类来源：

- `chat_message`
- `estimate_api`
- `manual`

#### 4.4.3 保存规则

- 只有用户主动保存时，才创建 Food Log。
- `chat_message` 来源必须带 `session_id`。
- `chat_message` 只能保存结构化估算结果。
- 文本回复与推荐回复不能直接保存到 Food Log。
- `/estimate` 成功响应也不会自动保存，仍需显式调用保存接口。

#### 4.4.4 幂等与去重

- Food Log 支持 `idempotencyKey`。
- 数据库对 `(user_id, idempotency_key)` 建唯一索引。
- Chat 保存默认使用消息相关 key。
- 独立 estimate 保存默认使用 `estimate_api:{clientRequestId}`。
- 如果命中已删除记录，系统会先恢复再更新，而不是新建重复条目。

#### 4.4.5 列表与检索

后端接口：

- `GET /food-logs`
- `GET /food-logs/{food_log_id}`
- `POST /food-logs`
- `POST /food-logs/from-estimate`
- `PATCH /food-logs/{food_log_id}`
- `DELETE /food-logs/{food_log_id}`
- `POST /food-logs/{food_log_id}/restore`

当前列表过滤能力：

- `sessionId`
- `dateFrom`
- `dateTo`
- `query`
- `sort`
- `limit`

当前排序能力：

- `created_desc`
- `created_asc`

当前状态判断：

- “优化 Food Log 的检索与筛选能力”最轻量级部分已完成。
- 现阶段已从仅按 meal 过滤，升级为支持关键词查询与排序。
- 更复杂的多维筛选仍未进入当前实现。

#### 4.4.6 编辑与生命周期

- 删除为软删除。
- 支持恢复已删除条目。
- 前端允许编辑食材克重，并按比例重算热量和三大营养素展示。
- 列表默认只返回 `active` 条目。

#### 4.4.7 图片与元数据

Food Log 当前已支持：

- `image`
- `imageSource`
- `imageLicense`

当前行为：

- 若有图片但未传来源，后端会给默认来源值。
- 若有图片但未传授权，后端默认 `user_owned`。
- 前端无图时展示占位图。
- 详情页展示来源与授权信息。

#### 4.4.8 标准菜品图片自动补齐（下一阶段需求）

产品目标：

- 为命中标准菜品的 Food Log 自动补齐统一、稳定的官方菜品图。
- 提升 Food Log 列表识别效率与整体产品质感。
- 建设可持续维护的“标准菜品视觉资产库”，而不是把图片能力绑定在单条 Food Log 上。

核心策略：

- 图片生成、存储与审核的粒度是 `standard_dish`，不是 `food_log`。
- 每个标准菜品只维护一张官方默认图；同一道标准菜品命中的所有 Food Log 复用同一张图。
- AI 生成只在标准菜品尚无图片时触发一次；审核通过后全局复用，除非人工重置，否则不重复生成。
- AI 生成结果不直接面向用户展示，必须先进入后台审核队列。

触发与复用逻辑：

- 当 Food Log 保存过程中，高置信命中标准菜品时，系统为该记录绑定 `standard_dish_id`。
- 若 `standard_dish.image_url` 为空，则异步触发 `image_generation_job`。
- 若 `standard_dish.image_status = approved`，则 Food Log 可直接复用该标准菜品图片。
- 若未命中标准菜品、命中置信度不足，或图片仍处于 `pending` / `null` 状态，则不阻塞保存，前端继续展示占位图。
- 若用户当前提问本身只得到低置信标准菜品候选，Chat 层应优先发起澄清追问，避免直接产出模糊菜品卡片并把不稳定条目写入 Food Log 或标准菜品资产链路。

统一生成规范：

- 所有标准菜品图必须使用统一的 Prompt Template 生成，禁止不同菜品使用分散、风格不一致的临时 prompt。
- 当前推荐模板要求包括：`菜品主体单一且居中`、`约 45 度轻微俯视角`、`米白/浅灰干净背景`、`柔和自然光`、`真实但略理想化`、`无人物/手/明显餐具干扰`、`无文字/水印/logo`、`适合作为产品卡片封面`。
- 统一 prompt 的目标是保证视觉资产库风格稳定，避免同类菜品出现“照片感、插画感、背景复杂度”明显不一致的问题。

审核与入库流程：

1. AI 根据统一模板生成标准菜品候选图。
2. 候选图进入后台审核队列，并写入候选记录或将标准菜品标记为 `pending`。
3. Admin 页面展示待审核列表，支持查看、通过、拒绝与重新生成。
4. 审核通过后，将图片标记为该标准菜品的官方图，并写回 `standard_dish.image_url` 与 `standard_dish.image_status = approved`。
5. 审核拒绝后，候选图保留审计状态或被丢弃，并允许重新发起生成。

数据结构建议：

- `standard_dishes`
  - `id`
  - `name`
  - `image_url`
  - `image_status` (`null / pending / approved`)
- `dish_images`（可选扩展表）
  - `id`
  - `dish_id`
  - `image_url`
  - `source = ai_generated`
  - `status = pending / approved / rejected`
  - `created_at`

治理要求：

- 标准菜品官方图属于受控产品资产，应继续保留来源与授权审计字段，沿用现有 `image / imageSource / imageLicense` 治理思路。
- Admin 审核通过前，候选图不得直接出现在用户侧 Food Log 列表或详情中。
- Admin 审核页面属于内部运营工具，只对管理员开放，不进入普通用户的信息架构、主导航或常规设置入口。
- 图片补齐是展示增强层，不改变估算结果，不参与识别主链路，也不影响 Food Log 主保存成功与否。

当前阶段不做事项：

- 不做开放式网络搜图或抓图后直接入库。
- 不允许用户上传图片参与标准菜品图建设。
- 不把图片作为菜品识别入口。
- 不承诺所有自由文本菜名都能自动补齐图片。

### 4.5 Profile

后端接口：

- `POST /profile`
- `GET /profile/me`
- `GET /profile/{profile_id}`
- `PUT /profile/{profile_id}`

当前字段：

- `age`
- `height`
- `weight`
- `sex`
- `activityLevel`
- `exerciseType`
- `goal`
- `pace`
- `kcalTarget`
- `dietStyle`
- `allergies`

当前业务规则：

- 一个用户只允许一个 Profile。
- 前端有完整 Profile 页面。
- 前端会根据年龄、身高、体重、性别、活动水平、目标、节奏自动计算推荐热量并回填 `kcalTarget`。
- 前端会本地缓存当前 Profile，并把 `profileId` 传入 Chat / Estimate。

Profile 对其他模块的影响：

- 推荐链路使用目标、热量目标、饮食风格、过敏原和活动信息。
- 估算链路会把 Profile 作为建议文案上下文。
- 过敏原会参与推荐安全拦截。

### 4.6 Insights

后端接口：

- `POST /api/insights/analyze`
- `GET /api/insights/history`
- `GET /api/insights/basket`
- `PUT /api/insights/basket`

#### 4.6.1 当前模型

Insights 当前已经从“临时实时分析”调整为“历史快照优先”的模型。

核心原则：

- 分析结果是显式生成的 snapshot。
- Food Log 后续发生变化时，不会自动重算已有分析。
- 用户需要通过“生成 AI 分析 / 重新分析”主动生成新结果。
- 历史结果优先于页面初始化时的实时分析。

#### 4.6.2 分析维度

当前支持：

- `day`
- `week`

分析输入：

- `mode`
- `dateRange`
- `selectedLogIds`（可选）
- `cacheKey`（前端可传）

分析输出：

- `aggregation`：总热量、蛋白、碳水、脂肪、比例、条目数
- `entries`：参与分析的简要条目
- `ai`：`summary / risks / actions`

#### 4.6.3 历史快照与唯一性

持久化层当前规则：

- `insights_analysis` 以 `user_id + mode + date_start + date_end` 唯一。
- 同一时间范围下重新分析，会覆盖旧结果。
- `selectedLogIds` 会被保存，但不会形成同一时间范围下的多版本并存。

这意味着：

- 当前版本的产品语义是“每个时间范围保留最新分析快照”。
- 它不是“同一天/同一周可保存多套筛选版本”的历史版本系统。

#### 4.6.4 页面加载逻辑

当前前端行为：

1. 先读取 `GET /api/insights/history`
2. 若当前时间范围已有历史结果，则直接展示
3. 若没有历史结果，则展示空状态并提供“生成 AI 分析”入口
4. 分析成功后更新本地缓存，并作为该时间范围的最新快照

#### 4.6.5 分析篮子同步

当前实现已支持分析篮子后端同步：

- 前端本地仍保留 basket 状态，保证交互即时性
- 同时通过 `/api/insights/basket` 做账号级同步
- 换设备或重新登录后，可恢复已同步的 basket 项

当前边界：

- basket 是“待分析条目选择状态”，不是正式分析历史
- 它没有版本比较或多人协作语义

#### 4.6.6 周分析展示状态

当前周分析已经具备日/周双模式、历史快照和 AI 文本输出能力，但仍不是完整的趋势分析产品。

现阶段更接近：

- 基础周视图
- 基础聚合结果
- AI 周期总结

尚未形成更强的趋势洞察能力，例如：

- 更长周期对比
- 多版本对比

## 6. 当前边界与已知限制

1. Chat 意图分流仍是规则驱动，复杂中文表达仍可能误判。
2. 推荐安全拦截仍是关键词级，不是结构化配料识别。
3. Chat 和 `/estimate` 的成功结果都不会自动写入 Food Log。
4. 独立 `/estimate` API 已实现，但没有正式主 UI 入口。
5. Insights 只保留同一时间范围下的最新快照，不保留多版本并存历史。
6. Food Log 已有基础检索与排序，但还不是完整的可组合高级筛选系统。
7. 图片能力当前已具备字段、治理与展示基础，但“标准菜品级 AI 图片生成 + 后台审核入库 + 全局复用”的闭环尚未进入主保存流程。
8. 当前项目已有首期中文食物知识库与检索增强，但仍属于种子库阶段，离“覆盖更全、标准化更强、持续运营的数据底座”还有差距。

- 同一标准菜品首次命中可触发一次生成任务；重复命中不再重复创建任务。

任务 10-C：Food Log 图片复用与用户侧回退逻辑

目标：

- 将审核通过的标准菜品官方图稳定接入 Food Log 列表与详情页，同时保持主流程非阻塞，并避免低置信模糊菜品污染数据底座。

范围：

- Food Log 保存时，在高置信命中标准菜品的前提下写入 `standard_dish_id`。
- 若用户输入仅命中低置信候选，则 Chat 必须先追问补足信息，不直接生成可保存卡片，也不写入模糊的 `standard_dish_id`。
- 若 `standard_dish.image_status = approved`，则在 Food Log 读取或写入阶段复用标准菜品官方图。
- 若未命中标准菜品、命中置信度不足，或图片仍为 `null / pending / rejected`，则继续使用占位图或空图展示。
- 普通用户不感知候选图状态，只看到“已有官方图”或“仍为占位图”的稳定结果。

验收标准：

- 同一道标准菜品在不同用户、不同 Food Log 中展示同一张官方图。
- 待审核与审核拒绝状态不会误展示到用户侧。
- 类似“炒面热量”这类模糊表达在未补足关键信息前，不会直接生成可保存的信息卡片，也不会沉淀为模糊标准菜品记录。
- 图片增强能力不改变 Food Log 的保存成功率、估算结果或既有编辑能力。

任务 10 的总体约束：

- 图片资产绑定在 `standard_dish`，不绑定在单条 `food_log`。
- 所有图片必须通过统一 Prompt Template 生成，保证构图、角度、背景、光线与风格一致。
- AI 结果先进入 `pending` 审核状态；审核通过前，用户侧继续显示占位图，不直接暴露候选图。
- Food Log 保存链路只负责标准菜品匹配与图片复用判断，不因图片生成或审核状态阻塞主保存流程。

任务 11：建设 Admin 菜品图片审核台（仅管理员可见）

目标：

- 为 AI 生成图片建立独立的后台审核入口，确保标准菜品官方图在入库前经过人工把关。

定位：

- 该页面是内部 Admin 工具，不属于普通用户产品功能，不出现在用户主导航、Explorer、Insights、Profile 或 Chat 入口中。
- 页面仅对管理员角色开放；非管理员不可访问对应路由、接口与图片审核动作。

范围：

- 提供待审核列表，至少支持按 `status`、创建时间、标准菜品名称进行浏览与筛选。
- 提供单条候选图详情，展示标准菜品名、候选图预览、生成时间、状态、可选 prompt 版本与最近操作记录。
- 支持 `approve`、`reject`、`regenerate` 三类基础操作。
- `approve` 后将候选图写为 `standard_dish` 官方图；`reject` 后保持候选图不可见；`regenerate` 重新进入生成流程但不覆盖当前已批准图片。
- 记录审核人、审核时间与操作结果，满足后续运营排查与审计需求。

验收标准：

- 管理员可以在独立后台完成候选图审核闭环，不需要直接修改数据库。
- 普通用户账号无法看到该页面，也无法调用对应审核接口。
- 审核动作完成后，标准菜品官方图状态与 Food Log 展示结果同步一致。

当前阶段明确不做：

- 不做开放式网络搜图。
- 不直接抓取第三方图片入库。
- 不开放用户上传图片作为标准菜品图片来源。
- 不把图片能力扩展为识别入口。

## 9. 结论

截至 2026-03-24，Food Pilot 已经不是纯 demo。

它已经具备：

- 真实账号体系
- 真实 Chat 会话与消息持久化
- 显式保存的 Food Log
- 可用的 Profile 个性化能力
- 历史快照优先的 Insights 分析闭环
- 首期面向中国用户的标准化食物数据底座

当前最需要继续收敛的不是“再加多少页面”，而是：

- Chat 规则分流的稳定性
- Insights 的分析表达与状态模型
- 独立 estimate 路径与 Chat 统一链路的工程收敛
- 标准菜品级 AI 图片生成、审核入库与 Food Log 自动补图闭环
