# FoodPilot 容器与分析对象数据模型规格

## 文档信息

- 文档主题：容器体系、最底层分析对象与扩展预留
- 更新时间：2026-04-05
- 关联文档：`docs/PRODUCT_SPEC.md`
- 适用范围：`Food Log / Explorer / Favorites / Insights / Chat decision cards`

---

## 1. 目标

这份文档用于定义 FoodPilot 下一阶段的“保存容器”和“分析对象”数据模型，解决三个问题：

1. 哪些对象可以保存
2. 哪些对象可以收藏
3. 哪些对象可以进入营养分析

核心目标不是把结构做复杂，而是把下面三层语义彻底分开：

- 容器
- 菜品对象
- 分析对象

只有这三层清楚，后续再扩入口、扩容器、扩平台时才不会推翻现有链路。

---

## 2. 设计原则

### 2.1 保存宽于分析

允许保存的对象范围，应大于允许进入 Insights 分析的对象范围。

### 2.2 容器不等于分析对象

容器只是收纳与组织结构，本身不能被当成营养分析输入。

### 2.3 最底层对象才可分析

只有最底层、稳定、单一的可食用对象，才能进入营养分析。

### 2.4 低置信不等于不可保存

来源不明确、品牌未知、自制菜品这类对象，可以先保存，但不一定立刻拥有正式归档资格。

### 2.5 预留扩展位

本模型必须为未来可能出现的新容器、新入口和新对象类型留出空间，避免写死成只适用于当前网站。

---

## 3. 核心分层

建议将数据对象分成四层：

1. `商品目录层`
   对应公共商品、私有商品、品牌、品类等稳定对象。

2. `保存容器层`
   对应 Explorer / Food Log 内用户看到的收纳结构。

3. `记录项层`
   对应一次实际保存下来的记录项。

4. `分析对象层`
   对应可被 Insights 直接消费的最底层可食用对象。

这四层不是同义词，不能混用。

---

## 4. 容器类型

### 4.1 正式分类容器

用途：

- 承接已经具备稳定分类标签的对象
- 与分层商品目录对齐

典型结构：

- 品类容器
- 品牌容器
- 品牌商品容器
- 私有商品容器

适合对象：

- 品类明确
- 品牌明确或用户私有商品明确
- 可稳定归档

### 4.2 未明确来源容器

用途：

- 承接可以估算、可以保存，但暂时无法稳定归到正式商品目录的对象

典型场景：

- 用户不知道品牌
- 家里自制菜
- 小店菜品
- 只知道菜名，不知道具体来源

关键语义：

- 允许保存
- 允许后续补信息
- 不强行映射正式品牌商品

### 4.3 收藏夹容器

用途：

- 沿用现有收藏逻辑
- 作为“复用、回看、比较、再次决策”的准备区

关键语义：

- 可收藏
- 可回看
- 可用于后续重新分析
- 不等于正式营养分析输入容器

### 4.4 扩展容器预留

未来允许新增其他容器类型，例如：

- `decision_history`
- `meal_plan`
- `shopping_candidates`
- `shared_list`
- `campaign_bucket`

因此容器类型不应写死为枚举后无法扩展，建议保留：

- 稳定主类型
- 可扩展子类型
- UI 可识别但后端不必一开始就全部实现

---

## 5. 容器字段建议

建议所有容器至少具备以下字段：

```ts
type ContainerType =
  | 'formal_catalog'
  | 'unknown_source'
  | 'favorites';

interface ContainerRef {
  containerId: string;
  containerType: ContainerType;
  containerSubtype?: string | null;
  title: string;
  parentContainerId?: string | null;
  userId: string;
  status: 'active' | 'archived' | 'deleted';
  sortOrder?: number | null;
  metadata?: Record<string, unknown>;
}
```

字段解释：

- `containerType`：主语义类型
- `containerSubtype`：预留未来扩展
- `parentContainerId`：支持容器分层
- `metadata`：预留平台、来源、实验字段等

---

## 6. 记录项与容器关系

### 6.1 记录项不是容器

一条 Food Log 记录本质上是“某次保存的结果项”，不应直接等同于容器。

### 6.2 一条记录项应绑定一个主容器

建议每条记录项至少有一个主归属容器：

- 正式分类容器
- 未明确来源容器

### 6.3 收藏应单独建关系

收藏不建议直接通过“复制一条记录到另一个列表”实现，而建议单独保留收藏关系。

原因：

- 收藏不是第二份记录
- 收藏可能跨多个正式记录复用
- 收藏与分析链路的语义不同

---

## 7. 最底层分析对象

### 7.1 定义

最底层分析对象，指一个可被 Insights 直接消费的单一、稳定、具备营养快照的可食用对象。

它必须满足：

1. 是单个具体食用对象，而不是上层容器
2. 有明确热量和宏量营养素快照
3. 不是“待进一步拆解”的模糊集合

### 7.2 可进入分析的例子

- 一份明确的“番茄炒蛋”
- 一杯明确的“瑞幸生椰拿铁”
- 一个套餐中拆出的“薯条”
- 一条“品牌未知但已确认是单份家常番茄炒蛋”的对象

### 7.3 不可直接进入分析的例子

- 一个品牌容器
- 一个收藏夹
- 一个“未明确来源容器”
- 一条尚未确定是一道菜还是两道菜的模糊记录

---

## 8. 分析准入字段

建议在最底层候选对象上加入明确准入字段：

```ts
interface AnalysisEligibility {
  analysisEligible: boolean;
  reasonCode:
    | 'eligible'
    | 'missing_nutrition_snapshot'
    | 'not_leaf_item'
    | 'ambiguous_object'
    | 'container_only'
    | 'needs_clarification';
  reasonDetail?: string | null;
}
```

这样做的价值：

- 前端可直接展示“为什么现在不能加入分析”
- 保存和分析逻辑可以分开
- 后续扩展原因码时不需要重写链路

---

## 9. 聊天卡片与容器映射

聊天中的决策卡片在下一阶段必须具备容器映射能力。

建议字段：

```ts
interface DecisionCardClassification {
  categoryId?: string | null;
  categoryName?: string | null;
  brandId?: string | null;
  brandName?: string | null;
  productId?: string | null;
  productName?: string | null;
  productScope: 'public' | 'private';
  itemRole: 'main' | 'drink' | 'side' | 'combo_child';
  containerType: 'formal_catalog' | 'unknown_source' | 'favorites';
  saveContainerKey: string;
  analysisEligible: boolean;
}
```

映射规则：

1. 若品牌和商品稳定命中，优先映射到 `formal_catalog`
2. 若对象可估算但来源不明确，映射到 `unknown_source`
3. 若用户主动收藏，则附加收藏关系，但不改变主保存容器

---

## 10. 收藏夹与正式记录的关系

建议收藏夹不要被设计成“另一套 Food Log”。

更合理的方式：

1. 收藏夹保存“对象引用”或“稳定快照”
2. 正式保存保存“记录项”
3. 同一对象可被收藏，也可有多次正式记录

这样可以支持：

- 同一个菜重复吃很多次，但收藏只保留一份
- 收藏用于回看和重选
- 正式记录用于分析和时间线

---

## 11. 容器迁移规则

必须允许对象在不同容器之间迁移。

典型迁移：

1. `unknown_source -> formal_catalog`
   当用户后续补齐品牌、做法或来源后，原先未明确来源的对象可迁入正式分类容器。

2. `formal_catalog + favorites relation`
   正式对象可被加入收藏夹，但不改变其主容器。

3. `unknown_source + favorites relation`
   未明确来源对象也可被收藏，用于后续回看。

迁移规则建议：

- 迁移不应破坏原始记录快照
- 迁移应保留审计信息
- 迁移应允许幂等重试

---

## 12. Insights 链路规则

### 12.1 准入

Insights 只能接收最底层分析对象。

### 12.2 不允许的输入

以下对象不能直接加入分析篮子：

- 容器
- 收藏夹整体
- 品牌整体
- 未明确来源容器整体

### 12.3 允许的输入

以下对象可以加入分析篮子：

- 容器内部某一具体菜品项
- 收藏夹里某一个具体菜品项
- 未明确来源容器里某一个已经稳定成单菜项的对象

### 12.4 UI 要求

UI 必须能明确区分：

- `保存`
- `收藏`
- `加入分析`

三者不是同一个动作。

---

## 13. 与未来扩展的兼容预留

为了给未来改动留空间，这套模型必须预留以下能力：

### 13.1 新入口兼容

未来入口可能来自：

- Chat
- OCR 截图
- 链接解析
- 浏览器插件
- 小程序
- 平台分享

因此容器和分析对象模型不能依赖某一个入口专属字段。

### 13.2 新对象类型兼容

未来可能出现：

- 一份套餐对象
- 一组比较候选对象
- 一周饮食计划对象
- 共享对象

因此要预留：

- `containerSubtype`
- `metadata`
- `reasonCode`

### 13.3 新分析模式兼容

未来 Insights 可能不仅分析日/周，还会分析：

- 品类偏好
- 品牌偏好
- 购买决策习惯
- 快乐餐频率

因此分析对象不能只保留热量字段，建议持续保留：

- 分类标签
- 来源范围
- 对象角色
- 时间快照

---

## 14. 推荐落地顺序

建议按以下顺序落地：

1. 先定义 `containerType`、`saveContainerKey`、`analysisEligible`
2. 再定义未明确来源容器和收藏夹容器
3. 然后把聊天卡片、保存逻辑和 Explorer 视图对齐
4. 最后再改 Insights，只允许最底层对象进入分析

---

## 15. 最小验收标准

满足以下条件即可认为容器模型最小成立：

1. 用户可以把来源不明确但可估算的菜品保存到 `未明确来源容器`
2. 用户可以继续使用 `收藏夹容器`
3. 聊天卡片能明确显示保存去向和是否可加入分析
4. 容器本身不能直接加入 Insights
5. 只有最底层菜品对象可以加入 Insights
6. 后续若增加新容器类型，不需要推翻现有主结构
